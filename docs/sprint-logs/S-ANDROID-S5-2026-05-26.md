# Sprint Log: S-ANDROID-S5

## Date Range
- Start: 2026-05-26
- End: 2026-05-26 (same-day continuation from S4)

## Objective
- Primary goal: Put the operationally-critical at-a-glance numbers on the operator's home screen so they can sanity-check the trader without opening the app. Status pill + 24h P&L + open trade count, refreshed every 15 min.
- Secondary goals:
  - Build the widget as a `Glance` `@Composable` (not legacy RemoteViews-by-XML) so the visual palette stays in sync with the in-app Status screen by construction.
  - Lay the foundation for S6's bigger 4×5 widget — shared cached state means the bigger widget reads the same `/api/bot/stats` pull without a second fetch path.
  - Wire `SIGNAL_EMITTED` on the bot side so the M12 S4 reserved toggle becomes usable for operators who want each ICT detection mirrored to their phone.

## Tier
- Tier 1 across both repos.
  - Bot side: one new `publish_event(SIGNAL_EMITTED, …)` call inside the existing `signal_audit_logger.log_signal` writer, gated to buy/sell rows (matches `/api/bot/signals` filter). Observer side-channel; the publish call is best-effort and wrapped — never propagates into the audit-writer path.
  - Android side: new widget package + WorkManager periodic worker. Read-only consumer of `/api/bot/stats` (an endpoint that already ships).
- Justification: live trader unchanged. The notifier is observer-only. The Android app is out-of-tier from the money-at-risk surface.

## Starting Context
- Active roadmap items: M12 (Android Companion App). S4 closed earlier same day (Notifications screen + per-kind channels + bot taxonomy).
- Prior sprint reference: [S-ANDROID-S4](S-ANDROID-S4-2026-05-26.md).
- Known risks at start:
  - WorkManager has a 15-min minimum for periodic work; finer cadences silently up-clamp.
  - Glance's coroutine context inside `provideGlance` doesn't play well with DataStore's Flow-based reads — needs a sync-feel state read for the composable, which means SharedPrefs (not DataStore) for the widget's cached state.
  - Adding `androidx.work:work-runtime-ktx` brings in `WorkManager`'s automatic `WorkManagerInitializer` content provider. Bundled correctly with Gradle's `androidx.startup:startup-runtime` pull-in transitively.
  - Signal volume is unbounded — the operator might be surprised by the push rate once they toggle `signal_emitted` on. The Notifications screen description ("Each ICT detection — reserved, lands in M12 S5") was deliberately worded so the operator goes in eyes-open.

## Work Completed
- **Android PR [`ict-trader-android#14`](https://github.com/benbaichmankass/ict-trader-android/pull/14)** — `feat(s5): Glance home-screen widget`. 3×2 default, resizable. Three new files under `app/widget/`:
  - `StatusWidget.kt` — `GlanceAppWidget` + `Receiver`. Composable layout reuses the Status screen's pnl + status-pill palette via `GlanceTheme` + `ColorProvider` so a future palette change doesn't have to be mirrored.
  - `StatusWidgetStateStore.kt` — SharedPrefs-backed cached state. Synchronous read inside `provideGlance` (DataStore's Flow would block the Glance coroutine).
  - `StatusRefreshWorker.kt` — `CoroutineWorker` driven by `WorkManager`. `ensureScheduled()` called from `IctApp.onCreate` with `ExistingPeriodicWorkPolicy.KEEP` so re-launching doesn't reset the cadence.
  - `res/xml/status_widget_info.xml` — `AppWidgetProviderInfo` with `updatePeriodMillis=0` (we drive refresh ourselves).
  - `AndroidManifest.xml` — `<receiver>` + provider meta-data.
  - `IctApp.onCreate` — added `StatusRefreshWorker.ensureScheduled(this)`.
- **Bot PR [`ict-trading-bot#2121`](https://github.com/benbaichmankass/ict-trading-bot/pull/2121)** — `feat(mobile_push): wire SIGNAL_EMITTED at signal_audit_logger`.

## Risks and Follow-Ups
- Remaining technical risks:
  - **Signal-push volume.** Until an operator turns the toggle on for the first time, we don't know how many pushes per hour the live bot generates. If it's noisy enough to be annoying, the right fix is a Tier-2 server-side cooldown (one push per `{strategy, symbol, side}` per N minutes) rather than client-side filtering. Logged as an S5 follow-up.
  - **Widget pull-failure noise.** Each WorkManager run that fails to reach the bot will log a Crashlytics-tracked exception (via `runCatching`'s onFailure → no, actually it doesn't — `runCatching` just sets the outage flag). So failures stay silent. Confirmed by code reading.
- Remaining product decisions (Tier 3): none.
- Blockers for S6: none — the bigger widget builds on the shared cached state, no new bot dependency.
