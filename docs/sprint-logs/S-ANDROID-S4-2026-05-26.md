# Sprint Log: S-ANDROID-S4

## Date Range
- Start: 2026-05-26
- End: 2026-05-26 (same-day, continuing autonomously from S3 close-out)

## Objective
- Primary goal: Turn the previously-blind "all pushes, all the time" notifier into a per-event-kind subscription contract — operator can silence Telegram-mirror noise while keeping trade-close pings loud, and the choice survives across app upgrades.
- Secondary goals:
  - Pull the implicit event-kind strings (`"trade_closed"`, `"telegram"`) into a single canonical taxonomy module on the bot side so future kinds (`signal_emitted`, `health_concern`, `service_down`, `pnl_digest`) land in one place — not three.
  - Surface the taxonomy via `GET /api/bot/devices/event-kinds` so the Android UI renders from a single source instead of a hand-mirrored constant.
  - Use Android's per-NotificationChannel system settings as a *second* mute axis (the in-app toggles + the OS-level controls partition the same kinds the same way).

## Tier
- Tier 1 across both repos.
  - Bot side: docs / observer side-channel / one new read endpoint / a refactor of two call sites to use constants instead of raw strings. The notifier's subscription wire itself already shipped in S1, so no new write-path code lands here.
  - Android side: new sub-screen + new NotificationChannels + a write to one existing bot endpoint (`POST /api/bot/devices/register`, which already accepted a `subscriptions` field).
- Justification: live trader, web-api unit, systemd files, and `config/*.yaml` were all unchanged. The new endpoint is a Tier-1 read with no secrets. The Android app is out-of-tier from the money-at-risk surface.

## Starting Context
- Active roadmap items: M12 (Native Android Companion App). S3 closed earlier same day (Positions / Trades / Signals / Health folded into Status + FAD update-loop fix).
- Prior sprint reference: [S-ANDROID-S3](S-ANDROID-S3-2026-05-26.md).
- Known risks at start:
  - The bot side already accepted arbitrary strings as subscription kinds — typos would silently never match a publish ("toggle doesn't work three weeks later" bug class). The taxonomy + 400-on-unknown validation was the gate to close that.
  - Renaming an event-kind string is a breaking change for every already-installed app (`device_tokens.subscriptions` references the old string). Pin tests so a rename = loud test failure, never a silent break.
  - Android `NotificationChannel` creation must happen at process start, *before* any FCM message lands. Means the kind list needs to be statically available on the client too — can't depend on a successful `/event-kinds` fetch.

## Risks and Follow-Ups
- Remaining technical risks:
  - **Sync drift between the two `event_kinds` files.** A future kind added on the bot side but forgotten on the Android side means the OS-system-settings won't show a channel for it (the in-app screen does because it pulls from the API). Mitigation: when a kind goes from reserved to in-flight, the Android `NotificationKinds.ALL` needs to include it for OS-level channels to exist. Logged as a follow-up; consider gradle codegen in a future sprint.
  - **Empty-list semantics asymmetry.** The bot side treats `subscriptions=[]` as "subscribe to all" (default-permissive). The Android UI documents this in the "no kinds selected = back to default" info card, but a confused operator might still toggle every Switch off expecting "no pushes at all" and be surprised. The fix is a UI bias (a separate "mute all" master toggle); deferred.
