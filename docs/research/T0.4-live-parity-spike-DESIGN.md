## Why the forecast head is now the lead candidate (not the embedding)

The T0.1 live-parity memo scoped this same serve-time question for the *embedding*.
Since then the evidence moved the lead: the **forecast** is the stronger,
lower-risk thing to promote.

| | T0.1 embedding (32 dims) | **T0.4 forecast (6 `fc_*` cols)** |
|---|---|---|
| Lift @ shipped 0.005 (purged-CV) | +0.012 macro_f1, f1_vol ~flat (base-rate cliff) | **+0.012** macro_f1, **+0.012 f1_vol** — holds |
| Base-rate robustness | fades as volatile thins; −0.004 @ 0.006 | **robust** across 0.003/0.004/0.005 |
| ETH generalization | +0.058 @ 0.003, weak at production rate | **+0.029 @ 0.005** (stronger, at production) |
| Dimensionality / overfit surface | 32 raw dims | **6 interpretable, scale-free** cols |

So the head worth soaking live is a **BTC/ETH-15m vol-regime head with the `fc_*`
block**, and this memo specialises the live-parity mechanism to the forecast.

## The serve-time question is IDENTICAL to T0.1 — same answer

A `candidate` head is offline-only; promoting to `shadow` means it predicts on
every live bar, so its feature vector must be computed **live** and match the
offline one bar-for-bar (`train == serve` parity, guaranteed here by *reusing the
same pure functions*). The `fc_*` block is the same class of problem the
`tsfm_emb_*` block is: computing it live means running `chronos-bolt-tiny`, which
needs **`torch` + `chronos-forecasting`** — **trainer-side only** today
(`requirements-backtest.txt`); the live `requirements.txt` is LightGBM-only (models
ship as a text booster string, no heavy ML runtime). So the narrow question is
unchanged:

> **Where does the Chronos forward-pass run at serve time, and how does its output
> reach the live regime head — without putting `torch` on the money-box?**

The full option analysis (A install torch live / B sidecar / C ONNX / D
mirror-publish) is in [`T0.1-live-parity-DESIGN.md`](T0.1-live-parity-DESIGN.md) §
Options and is **not re-derived here** — the ranking is identical because the
dependency and the per-bar integration point (`src/runtime/regime_bar_scoring.py`,
fetch-gated + wall-clock-budgeted) are the same. **Recommendation stands: Option
D (trainer-computed, mirror-published) to run the shadow soak; hold Option C
(ONNX) as the graduation path if a later stage needs tick-exact serve.**

## Why the FORECAST is *easier* to serve than the embedding (three simplifications)

Specialising Option D to the forecast is strictly simpler than for the embedding:

1. **Native chronos output, deterministic.** `fc_*` comes from
   `pipe.predict_quantiles()` — what the model is *trained* to produce — and the
   6 columns are a fixed deterministic transform of `{q10,q50,q90}` (log-return
   space vs last close; see `ml/datasets/forecast_features.py`). There is **no
   seeded random-projection reduction** as with the embedding, so the
   trainer-vs-published parity check has **one fewer moving part** (no reduction
   seed to match).
2. **6 scale-free floats to publish, not 32 raw dims.** The mirror payload per
   `(symbol, bar)` is tiny and self-describing; the as-of merge into the live row
   is the same `cross_asset_live`-style late-bound, fail-permissive pattern.
3. **One publish job can serve BOTH `fc_*` and `tsfm_emb_*`.** They share the
   model, the window, and the connector fetch — so if the embedding is ever
   promoted too, it rides the *same* trainer job + mirror channel. The live box
   gains **zero** torch either way.

## The parity contract (what the publish job must guarantee)

The trainer's published `fc_*` for a bar must equal what an inline live compute
would produce. Bounded by construction if the publish job:

- fetches the window via the **same `connector_for_symbol`** the live scorer uses
  (same exchange/source/timeframe — BTCUSDT→Bybit, etc.), so the candle view
  matches;
- uses the **same `context_len` / `min_context` / horizon / `FORECAST_QUANTILES`**
  the offline block used (pin them in one shared config the job + block both read);
- reuses the **exact pure fn** `compute_forecast_feature_rows` (not a
  re-derivation) — the same discipline that gives every other shadow head free
  parity.

A one-shot **offline-vs-published diff test** (compute `fc_*` for the last N live
bars two ways — inline vs the published mirror row — assert bit-close) gates the
wiring step. Staleness is the only residual gap: the live head reads the
**most-recent published** `fc_*` (minutes-fresh at the mirror cadence), which is
exactly the offline as-of-carry semantics and is **acceptable for a 15m head in
shadow (observe-only)**.

## Proposed phased path (each step operator-gated / backtest-gated as noted)

1. **Rebuild + freeze the promotion-candidate head** at the production
   `vol_threshold=0.005` with the `fc_*` block, BTC-15m (and ETH-15m). The A/B
   numbers already exist at 0.005 (purged-CV); this pins the exact head artifact.
   *Offline; no live change.* **Tier-1.**
2. **Stand up the mirror-publish job (D):** a trainer-side scheduled task that,
   per live symbol at bar cadence, fetches the window via `connector_for_symbol`,
   runs `compute_forecast_feature_rows`, and publishes the latest `fc_*` row to
   `runtime_logs/trainer_mirror/`; add a `compute_live_forecast_row()` in the
   per-bar scorer that *reads* it and merges it (mirroring `cross_asset_live`).
   Gate on the parity diff test. *Observe-only wiring.* **Tier-1/2 (a new trainer
   timer + a read-only live merge; no order path touched).**
3. **Promote the head `candidate → shadow`** and soak — compare the live shadow
   track record to the offline expectation; watch score drift (KS/PSI via the
   existing `/shadow/drift`). **← Operator gate (shadow promotion).**
4. **Only on a confirmed soak:** decide `shadow → advisory` — the live-influence
   gate, where the head's `P(volatile)` would actually drive the regime router's
   vol axis for BTC/ETH cells (the `REGIME_ML_VERDICT_MODE=use` path that is
   already live for the *current* BTC advisory head). If tick-exact serve is then
   required, build the ONNX path (C). **← Operator gate (Tier-3, live-influence).**

## The exact remaining approvals (so nothing is ambiguous)

- **Step 2 (mirror-publish wiring)** — a new trainer timer + a read-only live
  feature merge. No order path, no `mode:`/`execution:` change. Ships as a normal
  reviewed PR; the *live* half (the scorer read) is observe-only. **Operator OK in
  chat (Tier-2-ish) before deploying the live-side read.**
- **Step 3 (candidate → shadow)** — the promotion gate. Autonomous-Claude may
  write the registry to `advisory` on the trainer, but the *decision* to soak this
  specific head live is the operator's. **Explicit operator approval.**
- **Step 4 (shadow → advisory)** — the money gate: the head's verdict influences
  real BTC/ETH order routing. **Explicit operator approval + a walk-forward A/B**
  in the regime-router harness (the same bar the current BTC vol-gate cleared).

## What this memo does NOT do

No live/config/unit/order-path change; no promotion; no new dependency added
anywhere. It is the **map** from a proven offline result to a shadow soak, so the
operator can approve step-by-step. The single highest-leverage next build (still
offline until the step-2 operator OK) is the **mirror-publish job + parity diff
test** — that is the one piece of new machinery the whole path needs, and it is
shared with T0.1 if the embedding is ever promoted.
