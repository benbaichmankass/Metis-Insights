# M19 T0.1 — Frozen-TSFM embedding A/B evidence (BTC 15m regime head)

> **Status:** ✅ RAN 2026-07-01 on the trainer VM. **Result: POSITIVE (modest,
> consistent).** Adding a frozen `amazon/chronos-bolt-tiny` embedding block lifts
> the BTC 15m regime head vs identical hand-engineered features under a clean,
> controlled A/B. Offline/candidate — no live-path change. This is *encouraging
> evidence*, not a promotion decision (see caveats). Design:
> [`ai-model-strategy-roadmap-2026-07-01.md`](ai-model-strategy-roadmap-2026-07-01.md).

## The question

T0.1 asks the cheapest version of the M19 "reads-everything" thesis: **does a
frozen, generic, pretrained time-series representation add edge over
hand-engineered features — for $0, CPU-only?** We test it on the BTC 15m
volatility-regime head (`regime_label` ∈ {range, volatile}).

## Method

- **Embedder:** `amazon/chronos-bolt-tiny` (9M params, CPU) as a **frozen feature
  extractor**. Per bar, the trailing 64-close window → `pipeline.embed()` →
  encoder embedding `(seq, d_model=256)` → mean-pooled over the patch axis →
  **256-dim** vector → **seeded Gaussian random projection to 32**
  (`tsfm_emb_0..31`; data-independent → leakage-free). Built at `stride=4` and
  as-of carried forward (`scripts/ml/build_embeddings.py`).
- **Data:** `market_features` **v003** (BTCUSDT 15m, 175,272 rows back to
  2021-07), built from `market_raw` v002 with `vol_window_n=20`,
  `forward_window_m=5`, default `vol_threshold=0.003`. The embedding side-stream
  produced **43,823** non-zero embeddings, carried forward to every row.
- **A/B (controlled):** both heads read the **same v003 dataset**, same
  `time_aware_holdout` (0.2), same hyperparameters (`class_weight
  volatile=28×`, recency `half_life=60d`, 200 iters). The *only* difference is
  the 32 embedding columns.
  - `btc-regime-15m-lgbm-base-v003` — base 7 features only (the control).
  - `btc-regime-15m-lgbm-emb-v1` — base 7 + `tsfm_emb_0..31` (the treatment).
- **Eval set:** n_eval = 35,054; support = 30,966 range / **4,088 volatile**
  (~11.7%) — *identical* between the two runs (the check that the A/B is clean).

## Result (clean, v003 vs v003)

| Metric | Control (base) | + TSFM emb | Δ (emb − base) |
|---|---|---|---|
| **macro_f1** | 0.5761 | **0.6172** | **+0.0411** |
| **f1_volatile** | 0.3759 | **0.4055** | **+0.0296** |
| f1_range | 0.7763 | 0.8288 | +0.0525 |
| accuracy | 0.6707 | 0.7342 | +0.0635 |
| weighted_f1 | 0.7296 | 0.7795 | +0.0498 |
| precision_volatile | 0.2413 | 0.2743 | +0.0330 |
| recall_volatile | 0.8505 | 0.7774 | −0.0731 |
| n_eval | 35,054 | 35,054 | — |
| support_volatile | 4,088 | 4,088 | — |

**Verdict — LIFT.** Both decisive metrics improve: `macro_f1` **+0.041** and
`f1_volatile` **+0.030**. The gain is *quality*, not just a recall/precision
trade — `precision_volatile` rises **+0.033** (the head trades a little volatile
recall for materially better precision, and nets higher f1). The frozen generic
embedding carries signal that survives the 256→32 random projection and that the
hand-engineered vol features do not already capture.

## Methodology note — a confound we caught and corrected

The **first** comparison run was `btc-regime-15m-lgbm-v2` (dataset **v002**) vs
the emb head (dataset **v003**). It showed a much larger apparent win
(`f1_volatile` 0.217 → 0.406) — **but it was invalid**: v002 and v003 were built
with different `vol_threshold`, so their `volatile` base rates differed ~4×
(v002 2.75% vs v003 11.7%). The models were graded on *different problems*. The
tell was `support_volatile` differing between the two runs (964 vs 4,088). The
lesson, worth carrying forward: **comparing across dataset versions silently
changes the label definition** — an A/B must fix the dataset and vary only the
features. The control manifest (`btc-regime-15m-lgbm-base-v003`) exists to
enforce that.

## Caveats (why this is evidence, not a promotion)

- **Single symbol / single timeframe** (BTC 15m). One head is not the fleet.
- **`time_aware_holdout`, not purged-CV.** A shadow-promotion gate needs the
  purged/embargoed walk-forward (`PurgedWalkForwardSplitter`) + drift bounds to
  rule out leakage/overfit. This run is a fast holdout.
- **v003's `vol_threshold=0.003`** (11.7% volatile) differs from the shipped v2
  head's threshold (~3.6%). A promotion path must rebuild v003 at the production
  threshold before the numbers transfer to the live head.
- **Random projection is a v0 reduction.** PCA / a learned (fold-safe) reduction
  might extract more — or the projection might be adding tolerable noise. Untested.
- **Modest magnitude (+0.04 macro_f1).** Whether it justifies putting the
  torch/chronos dependency on the live money-box (required for shadow, where the
  embedding must be computed live for train==live parity) is a real cost/benefit
  call — see "Next".

## Environment note (dependency lesson)

The trainer VM runs **Python 3.11.0rc1** (a 2022 release candidate). The default
`torch` (2.12.1+cu130) crashes on import there (`torch._dynamo` "Duplicate
dispatch rule for `intern`" polyfill bug); `TORCHDYNAMO_DISABLE` can't help (it's
an import-time crash). The working combo is **`torch>=2.4,<2.10` CPU-only**
(resolved to 2.9.1+cpu, installed via `--index-url
https://download.pytorch.org/whl/cpu`) + **`transformers>=4.40,<5`**
(chronos-forecasting 2.3.0 breaks on the transformers-5 lazy import) +
chronos-forecasting 2.3.0. These pins are recorded in `requirements-backtest.txt`.
A stable-Python trainer venv would remove the torch constraint (a future vm-ops
item).

## Next (proposed, NOT auto-started — operator picks)

Given a real-but-modest lift, the highest-value follow-ups, in order:

1. **Confirm before believing:** re-run under **purged-CV** + on **≥2 more
   symbols/timeframes** (ETH 15m, BTC 1h). A +0.04 on one holdout head is
   promising, not decisive.
2. **Try a richer reduction** (PCA / learned, fit past-only) vs the v0 random
   projection — cheap, may widen the gap.
3. **Apply embeddings where hand-features are weakest** — the *direction* head
   and the `conviction-meta` stacker (T0.3), where the base features carry less
   signal than they do for a vol-regime label. This is likely where a learned
   representation pays off most.

Only if 1 confirms and the lift holds does the shadow-promotion path (with its
live-VM dependency) become worth it.
