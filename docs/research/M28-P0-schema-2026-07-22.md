# M28 P0 — data-model schema (TradeThesis + event/signal/valuation stores) — 2026-07-22

> **Status: Tier-1 schema spec** (P0's second deliverable, per the M28
> design-of-record `M28-macro-value-speculation-DESIGN.md`). This is the **data
> contract** P1's feeds, the sleeve engine, and the backtest harness are all
> built against. No code ships from this doc — it fixes the tables, field types,
> enums, the thesis lifecycle, and the **point-in-time discipline** before any
> feed or executor is written.

## 0. Design rules that bind every table here

1. **Point-in-time, always.** Every row carries BOTH `observed_at` (UTC — when
   *we* learned the fact) and `as_of` / `effective_date` (the fact's own date).
   Every backtest join filters `observed_at <= decision_time` and takes the
   **latest un-revised** value as-of that instant. A revised macro print is a
   **new row** (new `observed_at`), never an update — so a backtest can never see
   a revision that didn't exist yet. This is the #1 correctness invariant of the
   whole milestone.
2. **Traceable.** Every input a thesis rests on is a row with a `source` +
   `source_url` (or artifact ref). A thesis is reconstructable and replayable
   purely from its linked rows.
3. **Free-sourced.** `source` values are constrained to the free stack (`fred`,
   `sec_edgar`, `bls`, `bea`, `treasury`, `fed`, `rss`, `market_data`,
   `llm_extractor`, `manual`); `bigdata` is allowed but optional.
4. **Storage split.** *Operational* sleeve state (theses, watched events, live
   signals) lives in **`trade_journal.db`** (the live sleeve reads/writes it each
   scan — same home as `prop_tickets` / `learning_progress`). *Heavy historical
   panels* (valuation series, fundamentals corpus) reuse the trainer-side
   **`corpus_store.py`** panel + a small on-VM live cache. Point-in-time rules
   apply to both.
5. **Observe-only first.** Every table is written in shadow before any of it
   drives a live order (house discipline); the executor is P3+/gated.

## 1. `TradeThesis` — the core object (table `macro_theses`, `trade_journal.db`)

One row per thesis. `watched_events`, `signals`, and geometry ride as JSON
columns (decoded by the reader), mirroring how `order_packages` stores
`signal_logic`/`meta`/`model_scores`.

| Column | Type | Notes |
|---|---|---|
| `thesis_id` | TEXT PK | `mth-<ulid>` |
| `created_at` / `updated_at` | TEXT (UTC ISO) | |
| `status` | TEXT | `draft \| active \| invalidated \| closed \| expired` (§1a) |
| `rationale` | TEXT | human-legible thesis statement |
| `world_view` | JSON | `{regime, macro_tilt, theme}` |
| `signals` | JSON | array of `signal_id` refs into `macro_signals` (§3) — the evidence |
| `valuation` | JSON | point-in-time valuation snapshot refs (§4) + the computed value read |
| `ta_context` | JSON | `{symbol_candidates[], setup, levels}` |
| `instrument` | JSON | `{symbol, venue, express_as ∈ {debit_vertical, spot, etf, future}}` |
| `direction` | TEXT | `long \| short` |
| `entry_plan` / `target` / `invalidation` | JSON | invalidation is **thesis-based** (a condition), not a tight price stop |
| `horizon_days` | INTEGER | expected time-to-play-out |
| `max_hold_until` | TEXT (UTC) | hard time barrier (calendar, not bars) |
| `watched_events` | JSON | array of `{event_id, on_outcome:{if→action}}` (§2) — the non-price machinery |
| `thesis_conviction` | REAL [0,1] | the M28 conviction → future `c_macro` |
| `conviction_provenance` | JSON | which signals/weights produced it |
| `grade` | JSON | `{llm_grade, calibration_bin}` (the S2 grader meta-label) |
| `account` | TEXT | `alpaca_options_paper` for the soak |
| `linked_order_package_id` | TEXT | set when the thesis places (via `execute_pkg`); null while observe-only |
| `close_reason` | TEXT | `target \| invalidation \| event_outcome \| time_barrier \| manual` |
| `realized_pnl` | REAL | filled at close (broker-cash for options, per the options-lifecycle path) |

### 1a. Thesis lifecycle (state machine)

```
draft ──(passes gates: conviction ≥ floor, risk ok, instrument selected)──▶ active
active ──(target hit)──────────────▶ closed(target)
active ──(invalidation condition)──▶ invalidated ──▶ closed(invalidation)
active ──(watched-event outcome → exit rule)──▶ closed(event_outcome)
active ──(max_hold_until passed)───▶ closed(time_barrier)
draft  ──(never activates before max_hold_until)──▶ expired
```

Observe-only mode logs the *would-be* transition to the soak (§5) without
placing/closing anything. Live mode (P3+, gated) drives `execute_pkg` /
options-lifecycle close.

## 2. Event store — `macro_events` + `thesis_event_links` (`trade_journal.db`)

The heart of "events we watch and make decisions based on their **results**."
Two tables: the calendar of events, and the per-thesis decision rules against them.

**`macro_events`** — one row per known event (scheduled or discrete):

| Column | Type | Notes |
|---|---|---|
| `event_id` | TEXT PK | `evt-<kind>-<date>-<entity>` |
| `kind` | TEXT | `fomc \| cpi \| nfp \| pce \| gdp \| earnings \| auction \| filing \| ruling \| other` |
| `entity` | TEXT | affected symbol/asset-class/company (nullable for pure macro) |
| `scheduled_for` | TEXT (UTC) | when it happens/publishes |
| `status` | TEXT | `scheduled \| resolved \| cancelled` |
| `expected` | JSON | consensus/prior (nullable): `{metric, consensus, prior}` |
| `realized_outcome` | JSON | filled on resolution: `{metric, actual, surprise, direction, text_ref}` |
| `resolved_at` | TEXT (UTC) | |
| `source` / `source_url` | TEXT | free source (BLS/BEA/Treasury/Fed/SEC/…) |
| `observed_at` | TEXT (UTC) | point-in-time (when we ingested it) |

**`thesis_event_links`** — the decision rules (a thesis ↔ event, many-to-many):

| Column | Type | Notes |
|---|---|---|
| `thesis_id` | TEXT FK | |
| `event_id` | TEXT FK | |
| `on_outcome` | JSON | rule: `[{if: <predicate over realized_outcome>, action: <A>}]` |
| `resolved_action` | TEXT | filled when applied: which `A` fired |
| `applied_at` | TEXT (UTC) | |

`action A ∈ {enter, add, trim, exit, flip, hold, extend}`. The **resolver**
(P2) ingests a `realized_outcome`, finds active theses linked to that event,
evaluates each `on_outcome` predicate against the outcome, and (observe-only
first) logs/executes the matched action. Predicates are explicit/auditable
(e.g. `surprise > 0`, `actual > consensus`, `direction == "hawkish"`).

## 3. Signals store — `macro_signals` (`trade_journal.db`)

Every traceable input (macro reading, news/filing extraction, TA read) is a row.

| Column | Type | Notes |
|---|---|---|
| `signal_id` | TEXT PK | `sig-<ulid>` |
| `ts` | TEXT (UTC) | signal timestamp |
| `observed_at` | TEXT (UTC) | point-in-time ingest |
| `source` | TEXT | free-stack enum (rule 3) |
| `source_url` | TEXT | link to the story/filing/series (traceability) |
| `extractor_id` | TEXT | which extractor/version produced it (`llm-extract-v1`, `fred-erp-v1`, …) |
| `claim` | TEXT | the structured claim |
| `entity` | TEXT | symbol/asset-class/company |
| `direction` | TEXT | `bullish \| bearish \| neutral` |
| `magnitude` | REAL [0,1] | strength |
| `confidence` | REAL [0,1] | extractor confidence (honest-null when unknown) |
| `event_ref` | TEXT | optional `event_id` this signal concerns |
| `raw_ref` | TEXT | hash/pointer to the raw text (dedup + audit) |

The LLM extractor (S1) writes here in the honest-null `screenshot_parse.py`
spirit: omit any field not supported by the text; never fabricate a number.

## 4. Valuation snapshots — `valuation_snapshots` (corpus panel + live cache)

Point-in-time asset-class valuation metrics (the "value" core, §1a of the
design doc). Heavy history lives in the `corpus_store.py` panel; a small on-VM
cache serves the live sleeve.

| Column | Type | Notes |
|---|---|---|
| `symbol` / `asset_class` | TEXT | the thing being valued (SPY, TLT, GLD, SLV, GDX, …) |
| `metric` | TEXT | `erp \| real_yield_10y \| breakeven_10y \| hy_oas \| ig_oas \| gold_silver_ratio \| term_premium \| …` |
| `value` | REAL | the metric value |
| `as_of` | TEXT (date) | the data's own date (the release/observation date) |
| `observed_at` | TEXT (UTC) | when we computed/ingested it |
| `source` | TEXT | `fred` (layer-1) / `sec_edgar` (layer-2 company fundamentals) |
| `inputs` | JSON | the raw series used (traceability, e.g. `{earnings_yield, DFII10}`) |

Layer-2 (single-name fundamentals from SEC EDGAR `companyfacts`) uses the same
table with company `metric`s (`pe`, `ev_ebitda`, `fcf_yield`, `earnings_surprise`,
`net_debt_ebitda`, …) — built in P1, exercised as the universe widens.

## 5. Observe-only soak — `macro_thesis_soak.jsonl` + `/api/bot/macro-thesis/soak`

Before anything is live, the sleeve writes one soak row per scan/decision:
`{ts, thesis_id, would_be_status, would_be_action, instrument, thesis_conviction,
signals[], watched_events[], reason}` — the same observe-first shape as
`allocator_soak` / `pairs_soak`. A read endpoint mirrors the existing soak
surfaces (Tier-1, file/DB-backed). This is how P3's mechanics are verified and
P4's backtest is reconciled against live before any real order.

## 6. Read API surface (future, Tier-1)

Mirroring the existing dashboard/Android read pattern:
`/api/bot/macro-thesis/{theses,signals,events,soak}` — file/DB-backed,
best-effort, newest-first. Wired when P2/P3 land; listed here so the schema and
the consumer contract are designed together.

## 7. What this unblocks

With these contracts fixed, P1 can build the feeds independently and in
parallel (each writes to a defined table): the FRED valuation feed →
`valuation_snapshots`; the SEC EDGAR path → `valuation_snapshots` (layer-2) +
`macro_events` (filings); the free release-calendar populator → `macro_events`;
the LLM extractor → `macro_signals`. The sleeve engine (P3) reads all four to
assemble a `TradeThesis`; the backtest harness (P4) replays them point-in-time.
