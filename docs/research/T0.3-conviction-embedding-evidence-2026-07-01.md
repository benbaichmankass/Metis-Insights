# M19 T0.3 — TSFM embeddings on the conviction-meta stacker (BTC slice)

> **Status:** ✅ RAN 2026-07-01 on the trainer VM. **Result: directionally
> positive but INCONCLUSIVE — the label wall, exactly as predicted.** Adding 8
> frozen-TSFM embedding columns to the conviction win-probability head on the
> BTCUSDT slice lifted macro_f1 **+0.039** — same sign/magnitude as the T0.1
> regime lift — but on a **20-row / 5-positive eval**, which is far too small to
> conclude anything. Offline/candidate — no live-path change. Follow-up to
> [`T0.1-embedding-followup-evidence-2026-07-01.md`](T0.1-embedding-followup-evidence-2026-07-01.md).

## The question

T0.1 proved frozen embeddings lift the **self-supervised** volatility-regime
heads (label = price-derived, 175k rows). T0.3 asks the harder question: do the
same embeddings help a head whose label is a **trade outcome** — the
`conviction_meta` win-probability stacker (`won = pnl > 0`)? Trade-outcome labels
are scarce (the "labels bind, not compute" wall), so this is where the
representation is least likely to pay off. We test on the data-richest single
symbol (BTCUSDT), reusing the existing T0.1 BTC-15m embedding side-stream.

## Method

- **Dataset:** `conviction_meta` family (v2 — the T0.3 wiring), one row per
  closed/filled/non-backtest BTCUSDT order package joined to its trade outcome,
  from the live-synced `trade_journal.db` (3,068 trades total). The BTCUSDT slice
  is **99 rows** (win-rate 0.354). The 8 `tsfm_emb_*` columns are as-of
  (past-only) carried from `datasets-out/embeddings/BTCUSDT/15m/v001` — **all 99
  rows received a real embedding** (join validated: `nonzero_emb 99`).
- **A/B (controlled):** one dataset serves both heads; the ONLY difference is the
  8 embedding columns.
  - `conviction-meta-base-btc-v1` — calibrated lenses (c_strat/c_setup/c_wr/c_reg)
    + context (regime/vol_regime/adx_14/direction/strategy_name). The control.
  - `conviction-meta-emb-btc-v1` — same + `tsfm_emb_0..7`.
- **Eval:** `time_aware_holdout` 0.2 → **n_eval = 20** (15 loss / **5 win**),
  identical support across both runs (clean A/B). LightGBM 200 iters, seed 42.

## Result

| Metric | Base (control) | + TSFM emb | Δ (emb − base) |
|---|---|---|---|
| **macro_f1** | 0.5611 | **0.6000** | **+0.0389** |
| accuracy | 0.6500 | 0.7000 | +0.0500 |
| **f1 (win class)** | 0.3636 | **0.4000** | +0.0364 |
| f1 (loss class) | 0.7586 | 0.8000 | +0.0414 |
| weighted_f1 | 0.6599 | 0.7000 | +0.0401 |
| precision (win) | 0.3333 | 0.4000 | +0.0667 |
| recall (win) | 0.4000 | 0.4000 | 0.0000 |
| n_eval | 20 | 20 | — |
| support (win) | 5 | 5 | — |

**Verdict — directionally positive, statistically INCONCLUSIVE.** The point
estimate moves the right way and by a magnitude (+0.039 macro_f1) strikingly
close to the regime-head lift. **But the eval is 20 rows with 5 positives** — the
round metric values (0.70, 0.80, 0.40) betray the granularity: one trade
reclassified is worth ~0.05 accuracy. A Δ of "one or two trades" is noise, not
signal. This neither confirms nor refutes that embeddings help the conviction
head; it confirms the **binding constraint is labels** (99 closed BTC trades),
precisely the T0.1 thesis.

## Why this is the label wall, not a null

The regime heads had 175,272 rows because their label is computed from price
(self-supervised). The conviction head has 99 BTC rows because its label is a
*realized trade outcome* — and the bot has only closed 99 BTC positions. No model
choice fixes that; only more trade history (or backtest-augmented labels) does.
So T0.3's honest status is **"pipeline validated, answer pending data,"** not
"embeddings don't help conviction."

## What was validated (the reusable asset)

- The T0.3 wiring works end-to-end: `conviction_meta` v2 as-of joins a per-symbol
  embedding side-stream, the build + train + eval pipeline runs, and the A/B is
  clean (identical support). This is reusable for any future embedding-into-
  outcome-head experiment.
- The live→trainer DB sync (`scripts/ops/sync_trainer_data.sh` →
  `data/trade_journal.db`) is the label feedstock path; the conviction corpus is
  on the **live VM**, not the trainer (the trainer holds price/market_raw for the
  self-supervised heads).

## Caveats

- **n_eval = 20 / 5 positive.** Dominant caveat — the result is not significant.
- **Single symbol.** BTC only; a multi-symbol join (per-symbol embedding streams)
  would roughly 10× the corpus but needs streams for MES/MGC/ETH/equities that
  don't all exist yet.
- **`time_aware_holdout`, not CV.** For ~100 rows, repeated/purged k-fold would
  use the data far better than a single 20-row holdout — a prerequisite for any
  believable number here.
- **Paper trades included.** `conviction_meta` doesn't filter paper, so the
  win-prob label mixes paper and real fills — fine for a representation probe,
  but a real promotion would revisit it.

## Next (proposed, operator picks)

1. **Do NOT invest further in raw embeddings on the conviction stacker yet** —
   the head is label-bound; spend returns once the trade corpus grows or via
   backtest-augmented labels (design § 4.5 already flags the v2 meta-model as
   data-bottlenecked).
2. **The higher-leverage path is indirect:** let the embedding value reach
   conviction through the **improved `c_reg` lens** — i.e. promote the
   embedding-enhanced *regime* head (T0.1, where the lift is real and
   leakage-safe); the conviction stacker then consumes its better score for free,
   no raw embedding dims required.
3. **If revisiting directly:** switch the eval to repeated/purged k-fold and build
   a **multi-symbol** embedding join to lift n out of the single-digit-positive
   regime before trusting any delta.

  `conviction-meta-emb-btc-v1` (one shared v001 embedded dataset; the base just
  doesn't select the `tsfm_emb` columns).

  --symbol-scope BTCUSDT --timeframe all -- db_path=data/trade_journal.db
  symbol=BTCUSDT embedding_path=datasets-out/embeddings/BTCUSDT/15m/v001`; train
  each manifest with `python -m ml train … --datasets-root datasets-out` on the
  trainer `.venv` (lightgbm 4.6.0).
