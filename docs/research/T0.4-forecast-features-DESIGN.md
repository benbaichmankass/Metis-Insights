## Thesis

T0.1 used `chronos-bolt-tiny` as a frozen **embedder** (representation). T0.4 uses
the **same** frozen model as what it's actually trained to be — a **quantile
forecaster** — and turns its predicted next-bar distribution into a handful of
**scale-free, leakage-safe features** a LightGBM head can consume. Where the
embedding carries a *volatility signature*, the forecast quantiles carry an
explicit, interpretable **forward view**: expected move, expected range, and
skew. The A/B question: do these lift the regime / conviction heads beyond the
hand-engineered features — and, importantly, do they help where the embedding
*didn't* (direction, and the rare-volatile regime)?

## Why this is a distinct bet from T0.1

- **Interpretable, low-dimensional.** 6 columns vs the embedding's 32 — far less
  overfitting surface, and each column *means* something (median return, range,
  skew). That matters on the label-bound heads (conviction) where 32 raw
  embedding dims swamped a ~100-row slice.
- **A forward view, not a state summary.** The forecast's predicted **range**
  (q90−q10) is a *forward* volatility estimate — plausibly more useful to a
  vol-regime head than a backward-looking realized-vol feature, and it does not
  depend on the volatile class being populated the way a learned representation
  does (the T0.1 base-rate cliff).
- **Reuses the exact same infra** — same model, same trainer-side dep, same
  as-of-join / candidate-stage / injectable-lazy-fn pattern. Near-zero new
  surface.

## The feature block (built this session)

`ml/datasets/forecast_features.py` — mirrors `embedding_features.py` exactly
(stdlib-safe import, injectable lazy `chronos_forecast_fn`, pure
`compute_forecast_feature_rows`, neutral-0.0-on-failure, `stride` + as-of carry).
Per strided bar it runs `pipe.predict_quantiles()` over the trailing close window
(past-only), takes the next-bar predicted price quantiles `{q10, q50, q90}`, and
emits 6 scale-free `fc_*` columns in log-return space vs the last close:

| Column | Meaning |
|---|---|
| `fc_ret_med` | median predicted log-return (directional expectation) |
| `fc_range_rel` | (q90−q10)/last_close — **forward volatility proxy** |
| `fc_up_prob` | sign proxy for P(up) (median > 0) |
| `fc_skew` | (q90+q10−2·q50)/(q90−q10) — forecast asymmetry |
| `fc_q10_rel` | q10 predicted log-return (downside) |
| `fc_q90_rel` | q90 predicted log-return (upside) |

Wired into `market_features` via an optional `forecast_path` kwarg (builder
v10→v11; 0.0 when omitted → existing builds unchanged), producer
`scripts/ml/build_forecasts.py`, candidate manifest
`btc-regime-15m-lgbm-fc-v1.yaml`, and a stub-based test suite (no torch needed).

## Results — the A/B (ran 2026-07-01, trainer VM)

**Method.** Built the `fc_*` side-stream on the trainer (`build_forecasts.py`
over BTC-15m `market_raw` v002, context-len 64, stride 4 → the as-of carry fills
the gaps), joined it into `market_features` via `forecast_path=…` at three
volatile base rates, and trained the BTC-15m **vol-regime** head **base (7
hand-engineered features)** vs **base + 6 `fc_*` columns** on the same dataset,
`time_aware_holdout` split. Δ = fc − base. Every arm fixes the dataset and varies
only the `fc_*` columns (the clean-comparison discipline from T0.1). The fc
side-stream is **fully populated** post-fix (`nonzero_fc 175272/175272` — see the
methodology note below on why that mattered).

| vol_threshold | volatile base rate | base macro_f1 | fc macro_f1 | **Δ macro_f1** | base f1_vol | fc f1_vol | **Δ f1_volatile** |
|---|---|---|---|---|---|---|---|
| **0.005** (shipped) | 4.6% | 0.5808 | 0.6037 | **+0.023** | 0.2169 | 0.2503 | **+0.033** |
| 0.004 | 8.4% | 0.5806 | 0.5991 | **+0.018** | 0.2760 | 0.2948 | **+0.019** |
| 0.003 | 16.1% | 0.5761 | 0.6028 | **+0.027** | 0.3759 | 0.3942 | **+0.018** |

**Verdict — LIFT, and it's base-rate-ROBUST.** The forecast features add a steady
**+0.018 to +0.027 macro_f1** and a **positive f1_volatile at every threshold**.
This is the headline contrast with T0.1: the *embedding* lift was a base-rate
cliff (+0.052 at 0.003 → +0.012 at the shipped 0.005 → −0.004 at 0.006 —
[`T0.1-embedding-followup-evidence-2026-07-01.md`](T0.1-embedding-followup-evidence-2026-07-01.md)
§ Threshold sensitivity), so it stopped helping exactly at the production
operating point. The **forecast features do not decay as the volatile class
thins** — at the shipped 0.005 they give **+0.023 macro_f1, ~2× the embedding's
+0.012**, with the biggest f1_volatile gain (+0.033) of the three. That is the
T0.4 thesis confirmed: a *forward* volatility estimate (the predicted q90−q10
range) is a better vol-regime feature than a *backward* learned representation,
and it doesn't depend on the class being populated the way the embedding did.

**Why the forecast helps where the embedding faded.** The embedding is a
mean-pooled *state summary* — it encodes "how the series has been behaving,"
which is only discriminative for the volatile class when that class is
well-populated (lots of examples to separate). The forecast's `fc_range_rel` is
an *explicit forward variance estimate* per bar — it carries signal about the
*next* bar's dispersion regardless of how rare volatile bars are in the label
set. Six interpretable columns vs 32 raw embedding dims also means far less
overfitting surface, which matters most at the sparse 0.005 rate.

### Methodology note — the zero-vector bug (why the first run was void)

The initial A/B was **invalid** and never reported: the producer called
`pipe.predict_quantiles(context=contexts, …)`, but chronos-bolt's first
positional arg is `inputs`, so every forecast raised
`missing 1 required positional argument: 'inputs'` and the block's
neutral-0.0-on-failure guard silently emitted a **zero vector for all 43,823
rows** — the "A/B" was comparing base against base-plus-zeros. Fixed in PR #5335
(pass the batch positionally), the side-stream rebuilt fully non-zero, and the
numbers above are the first *valid* run. The stub-based unit tests can't catch
this because the real-chronos lines are `# pragma: no cover`; a trainer-side
smoke test that runs one real forecast is the logged follow-up.
