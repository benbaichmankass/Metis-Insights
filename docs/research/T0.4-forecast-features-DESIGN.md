## Thesis

T0.1 used `chronos-bolt-tiny` as a frozen **embedder** (representation). T0.4 uses
the **same** frozen model as what it's actually trained to be — a **quantile
forecaster** — and turns its predicted next-bar distribution into a handful of
**scale-free, leakage-safe features** a LightGBM head can consume. Where the
embedding carries a *volatility signature*, the forecast quantiles carry an
explicit, interpretable **forward view**: expected move, expected range, and
skew. The A/B question: do these lift the regime / conviction heads beyond the
hand-engineered features — and, importantly, do they help where the embedding
*didn't* (direction, and the rare-volatile regime)?

## Why this is a distinct bet from T0.1

- **Interpretable, low-dimensional.** 6 columns vs the embedding's 32 — far less
  overfitting surface, and each column *means* something (median return, range,
  skew). That matters on the label-bound heads (conviction) where 32 raw
  embedding dims swamped a ~100-row slice.
- **A forward view, not a state summary.** The forecast's predicted **range**
  (q90−q10) is a *forward* volatility estimate — plausibly more useful to a
  vol-regime head than a backward-looking realized-vol feature, and it does not
  depend on the volatile class being populated the way a learned representation
  does (the T0.1 base-rate cliff).
- **Reuses the exact same infra** — same model, same trainer-side dep, same
  as-of-join / candidate-stage / injectable-lazy-fn pattern. Near-zero new
  surface.

## The feature block (built this session)

`ml/datasets/forecast_features.py` — mirrors `embedding_features.py` exactly
(stdlib-safe import, injectable lazy `chronos_forecast_fn`, pure
`compute_forecast_feature_rows`, neutral-0.0-on-failure, `stride` + as-of carry).
Per strided bar it runs `pipe.predict_quantiles()` over the trailing close window
(past-only), takes the next-bar predicted price quantiles `{q10, q50, q90}`, and
emits 6 scale-free `fc_*` columns in log-return space vs the last close:

| Column | Meaning |
|---|---|
| `fc_ret_med` | median predicted log-return (directional expectation) |
| `fc_range_rel` | (q90−q10)/last_close — **forward volatility proxy** |
| `fc_up_prob` | sign proxy for P(up) (median > 0) |
| `fc_skew` | (q90+q10−2·q50)/(q90−q10) — forecast asymmetry |
| `fc_q10_rel` | q10 predicted log-return (downside) |
| `fc_q90_rel` | q90 predicted log-return (upside) |

Wired into `market_features` via an optional `forecast_path` kwarg (builder
v10→v11; 0.0 when omitted → existing builds unchanged), producer
`scripts/ml/build_forecasts.py`, candidate manifest
`btc-regime-15m-lgbm-fc-v1.yaml`, and a stub-based test suite (no torch needed).
