# T1.1 follow-up — TCN label-sensitivity arms (vol_threshold 0.003 / 0.004): evidence (2026-07-06)

**Question** (follow-up logged in `MB-20260703-001` / the T1.1 sprint): was the
TCN's clean negative at the production label (`vol_threshold=0.005`, ~3.6%
volatile base rate) **label-sparsity-driven** — too few positive gradients per
epoch for a deep model — rather than "depth has no value here"?

**Answer: the sparsity hypothesis is supported.** Both arms trained clean on
rented pods (same TCN architecture, same purged 5-fold CV, same 87,605 eval
bars; the only change is the label threshold in the on-pod `market_features`
build, threaded via `dataset.build_params`, #5635 + #5645). f1_volatile and
macro_f1 rise **monotonically** as the volatile class densifies:

|---|---|---|---|---|---|---|---|
| TCN @ 0.005 (2026-07-02 baseline) | 0.005 | 3.6% | 0.162 | — | — | 0.534 | 0.835 |
| TCN vt004 (`btc-regime-15m-tcn-vt004-v1`) | 0.004 | 7.0% | 0.282 | 0.197 | 0.506 | 0.590 | 0.823 |

## Honest bounds

1. **f1 at higher prevalence is mechanically easier** — the monotone rise
   alone does not prove the TCN "got better"; part of it is the metric's
   prevalence sensitivity. What it DOES establish is that the 0.005 negative
   cannot be read as "depth has no value on this data" without the sparsity
   confound.
2. **The tree control at the same label is the decider.** At matched ~14%
   prevalence the vt003 TCN reads above the LightGBM *registry-row* numbers
   (f1_volatile 0.423 vs 0.390, macro_f1 0.635 vs 0.560), but that registry
   row came from a different eval protocol/window — suggestive, NOT a
   controlled A/B (the same caveat the T1.1 sprint recorded). The clean
   control — `classification_lightgbm` trained on a `market_features` build
   at vol_threshold 0.003, identical purged CV — is a cheap CPU train on the
   trainer; run it next session before drawing any deep-vs-tree conclusion.
3. **Neither arm is promotion-eligible** (stated in both manifests): a
   sensitivity probe at a non-production label. If the tree control confirms
   a genuine TCN edge at denser labels, the *decision* that follows is a
   Tier-3 conversation about the label definition itself (what "volatile"
   should mean for the regime gate), not a promotion of these probes.
4. Both model bundles are GitHub Actions artifacts
   (`gpu-burst-model-28773344221` / `-28773756599`, 30-day retention); the
   trainer-side ingest failed on the box's SSH pressure (chronos build) —
   re-ingest from the artifacts or retrain trainer-side if registry entries
   are wanted.
