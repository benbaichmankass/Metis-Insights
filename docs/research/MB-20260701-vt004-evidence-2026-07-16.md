# MB-20260701-001 — BTC-15m vol-regime head at a lower vol_threshold: purged-CV first-gate evidence (2026-07-16)

## Question

The **live** BTC-15m advisory vol-regime head (`btc-regime-15m-lgbm-v2`) labels
"volatile" at `vol_threshold=0.005` (~3.6–4.6% base rate) where it is
**data-starved**: purged-CV `f1_volatile` ≈ 0.24 (recall ~0.51, precision ~0.16).
The M19 T0.1 sweep (trainer #5311/#5320/#5321) suggested a lower threshold is
materially more discriminative. This head's `vol_threshold` **defines the live
vol-gate's "volatile" operating point** (`REGIME_ML_VERDICT_MODE=use`, BTC enforce
is LIVE), so it is **Tier-3**: evaluate + propose, do not change unilaterally.

This is the decider on **current data** under the robust protocol: a
live-faithful mirror of the lgbm-v2 head (same 7 features, same 60d recency
half-life) with the volatile label built at a lower threshold, evaluated under
**purged 5-fold walk-forward CV** (not the live head's `time_aware_holdout`).

## Run

- Manifest: `ml/configs/btc-regime-15m-lgbm-vt004-pcv-v1.yaml` (candidate stage,
  offline — refused by the live shadow factory).
- Dataset: `market_features` BTCUSDT/15m built with
  `dataset.build_params.vol_threshold: 0.004`.
- Eval: `purged_walk_forward`, `n_folds=5`, `embargo_fraction=0.01`,
  `label_horizon=5`, `min_train_fraction=0.5`.
- Trainer: `LightGBMMulticlassTrainer`, `class_weight.volatile=11.9`,
  `sample_weight.half_life_days=60`, `n_iter=200`, seed 42.
- Ran on the trainer VM (pid 63111), completed 2026-07-16 ~17:56Z.
  n_train_final 175,272 / n_eval 87,636.

## Verdict: POSITIVE FIRST-GATE — but with a base-rate caveat that blocks a clean threshold claim

**What is confirmed (the backlog thesis):** at a *denser* volatile label the
head is **far more discriminative** than at the shipped 0.005 operating point —
`f1_volatile` 0.44 vs ~0.24, `macro_f1` 0.63, recall 0.77 (vs ~0.51), on a large
leakage-safe purged-CV sample. The shipped head is genuinely data-starved at 0.005;
a denser label produces a head that actually separates the classes. macro_f1 0.63
is healthy (not bought by crushing the minority class — recall_volatile is 0.77).

**What this run does NOT cleanly establish — the honest caveat.** This build was
parameterized `vol_threshold=0.004`, yet it produced a **14.05% volatile base
rate**. That is **inconsistent with the T1.1 track's threshold→base-rate mapping**
(measured on the same `market_features` family): T1.1 found 0.004 → **7.0%** and
0.003 → **14.0%** (`MB-20260703-001` evidence, issues #5647/#5650/#5658). This
run's 14.05% matches the T1.1 **0.003** label, not 0.004 — and its f1_volatile
0.438 lines up with the T1.1 **matched-label LightGBM control at 0.003** (0.4441,
#5669), not a distinct 0.004 point.

So the two tracks disagree on what base rate a given `vol_threshold` yields —
likely a **different `vol_threshold` semantic or a different dataset window/version**
between the `market_features` build path here and the T1.1 arms. Until that is
reconciled, this run is honest evidence that **"a denser volatile label makes the
BTC-15m head materially more discriminative"** (thesis confirmed, consistent with
both tracks at ~14% prevalence), but it is **NOT** a clean measurement that
"0.004 specifically is the right operating point" — because the "0.004" here
behaved like 0.003.

## Remaining gates before ANY Tier-3 config change (do NOT flip the live threshold)

The live threshold stays 0.005. Before proposing an operating-point change:

1. **Reconcile the threshold→base-rate mapping.** Rebuild the vol-regime
   `market_features` at 0.005 / 0.004 / 0.003 through the *same* build path and
   confirm the base rates, so a chosen threshold has a known, reproducible
   volatile prevalence. (The current cross-track disagreement — 0.004→7% in T1.1
   vs 0.004→14% here — must be resolved first; it may be a build-param semantics
   difference.)
2. **RG4 live-regime discrimination** (`scripts/ml/rg4_targeted.sh`) — does the
   denser-label head actually discriminate live volatile episodes on a fresh
   mirror (train/serve-skew gate), not just offline CV.
3. **Vol-gate backtest A/B** — the head operates the live gate
   (`REGIME_ML_VERDICT_MODE=use`); a threshold/head change must beat the shipped
   0.005 gate on a walk-forward money A/B (the same bar the 2026-06-27 vol-gate
   go-live cleared), not just on classifier metrics.
4. **Operator approval** — Tier-3, order-routing-affecting.

## Bottom line

The backlog's core claim is **validated**: the shipped 0.005 head is data-starved
and a denser volatile label yields a materially better classifier (f1_volatile
0.44 vs 0.24). This is a real, positive first-gate result and justifies pursuing
a denser operating point for the vol-gate head. But the **specific threshold is
not yet pinned** (the base-rate cross-track disagreement), and three Tier-3 gates
(mapping reconcile → RG4 → vol-gate backtest A/B → operator) stand between this and
any live change. No config was touched.
