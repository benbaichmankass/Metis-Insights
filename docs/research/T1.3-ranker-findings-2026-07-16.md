# T1.3 — Cross-strategy learned net-R ranker: findings (2026-07-16)

**Verdict: HONEST NEGATIVE.** A learned cross-strategy P(win) ranker has real,
per-fold-robust out-of-sample ranking AUC (0.61 market-only, up to 0.68 with
owner/cell features), but that AUC **does not convert to a portfolio selection
edge** in the sizing-normalized concurrency-capped backtest: the learned-ranker
selection arm is worse than both dumb symbol-priority (−$39) and the existing
rules-based EV scorer (−$211). The M18 learned-ranker-as-selector track is
CLOSED; the rules EV scorer stays the selector, and M18 P2/P3 (allocator
*selects* the subset) stays PARKED. This confirms the 2026-06-30 M18 P1 thesis
(pooled AUC ≈ 0.51, selection ≈ priority) with a properly-trained ranker and a
richer feature set — the conclusion holds.

## What T1.3 asked

Does a learned model that ranks the full cross-strategy candidate batch by
expected net-R let a capital allocator pick a better subset (under a shared
concurrency cap) than the current dumb symbol-priority order? This is the
`MB-20260629-ALLOC-RANKER` question and the substance of ROADMAP row T1.3.

## Method (all Tier-1 research, leakage-controlled)

1. **Candidate dataset** — `scripts/research/allocator_candidate_dataset.py`
   built 2,334 labelled candidates over BTCUSDT+ETHUSDT+SOLUSDT (5m clock,
   full symbol-family roster; single-position-per-symbol; fixed-notional risk
   so `net_r` is balance-free). Features are all decision-time / past-only:
   confidence, ev_r, rr, stop/tp dist, ret_1h/4h/12h, vol_1h, mom_align_1h,
   hour, dow, the (trend,vol) regime label, and per-(owner,regime)-cell
   historical expectancy (`cell_hist_*`, folded in at close → leakage-free).
2. **Walk-forward ranker AUC** — `scripts/research/allocator_ranker_eval.py`,
   expanding-window 5-fold, reporting pooled **and per-fold** OOS AUC.
3. **Selection backtest** — `scripts/research/allocator_multisymbol_backtest.py`
   with the new `--ranker-csv` learned arm: a frozen market-only P(win) ranker
   fit on the earliest 60% of candidates (before 2024-04-13), driving contested
   concurrency-slot selection over the strictly-later OOS window (all arms
   evaluate the same window → leakage-free). Compared against the EV scorer arm
   and the EV-blind symbol-priority control.

## Result 1 — the AUC gate PASSES (necessary, not sufficient)

Walk-forward pooled OOS AUC + per-fold (oos_n=1867):

| Feature stack | pooled AUC | per-fold AUC | fold-min |
|---|---|---|---|
| market-only | 0.611 | 0.648 / 0.612 / 0.555 / 0.624 | **0.555** |
| +regime | 0.616 | 0.629 / 0.634 / 0.568 / 0.630 | 0.568 |
| +cell-expectancy | 0.646 | 0.667 / 0.653 / 0.591 / 0.663 | 0.591 |
| +regime+cell | 0.644 | 0.643 / 0.662 / 0.604 / 0.658 | 0.604 |
| +owner | 0.678 | 0.714 / 0.693 / 0.610 / 0.686 | 0.610 |
| +regime+cell+owner | 0.680 | 0.703 / 0.703 / 0.624 / 0.682 | 0.624 |

Single-feature baselines (whole sample): ev_r 0.416, confidence 0.434, rr 0.445,
ret_1h 0.441, cell_hist_mean_r 0.553, cell_hist_winrate 0.648.

Every stack clears OOS AUC ≥ 0.55 **on all four folds** — no single-fold
knife-edge. This is materially better than the 2026-06-30 M18 P1 probe (pooled
≈ 0.51). The richer feature set + the full cross-strategy candidate set do carry
real cross-candidate ranking signal; the strongest signal is the per-cell
historical expectancy (`cell_hist_winrate` alone AUC 0.648) and owner identity
(the owner dummies encode which strategy legs bleed — `trend_donchian_eth`/
`trend_donchian_sol` have strongly negative weights).

## Result 2 — the selection backtest FAILS (the decisive gate)

Sizing-normalized 4-arm backtest, OOS 2024-04-13 → now (229,159 5m bars, shared
`--max-concurrent 2`, ranker fit on 1,400 pre-split rows, market-only):

| Arm | net PnL | maxDD % | trades | WR % |
|---|---|---|---|---|
| independent (per-symbol baseline) | −62.5 | 3.97 | 878 | 24.4 |
| ev (EV scorer selection) | −374.9 | 10.1 | 713 | 24.7 |
| shared_priority (EV-blind control) | −546.7 | 12.1 | 733 | 23.9 |
| **learned (ranker selection)** | **−586.0** | 11.8 | 734 | 23.7 |

Selection deltas (same budget engine + cap → isolates SELECTION):

- **learned − shared_priority = −$39.4** → `learned_beats_priority_net = false`
  (maxDD ~flat, −0.28pp).
- **learned − ev = −$211** → `learned_beats_ev_net = false`.
- (For reference: ev − shared_priority = **+$171.8**, maxDD −2.0pp — in this
  window the *rules* EV scorer's selection does beat priority.)

The pre-registered gate — `learned_beats_priority_net = true` AND positive net
delta AND no maxDD regression — is **not met**. The learned ranker's selection
is worse than dumb priority and much worse than the incumbent EV scorer.

## Why the AUC doesn't convert

Pooled/per-fold AUC measures "rank a random winner above a random loser across
*all* candidates." The allocator's real job is different: pick the best among
the 2–3 *simultaneous cross-symbol* candidates competing for a scarce slot at a
tick. Much of the ranker's AUC lives in **between-owner base-rate** separation
(cell-expectancy + owner identity — "this strategy leg is bad") rather than a
**within-tick** discriminator, and the market-only P(win) differences among
co-occurring candidates are small/noisy relative to the EV scorer, which folds
in RR and cost directly and is the more decision-relevant quantity. Note also
that every arm is net-negative over this window — the opportunity set itself was
a losing book, so "good selection" only loses slightly less, and the independent
per-symbol baseline (−$62) beats every shared-budget arm (the shared cap hurts
here). None of that is an argument for a learned ranker.

## Decision

- **Keep the rules EV scorer** (`allocator_ev.candidate_ev_score`) as the
  allocator's selection function and the shadow baseline. It is the arm that
  actually beats priority in this window.
- **Close the learned-ranker-as-selector track** (`MB-20260629-ALLOC-RANKER`):
  the ranker has real AUC but no selection alpha at this data scale / feature
  set. Honest negative recorded per the item's own resolution criteria.
- **M18 P2/P3 (allocator selects the subset) stays PARKED** — no proven within-
  tick P_win input to graduate it. Routing is unchanged; the allocator remains
  observe-only (`allocator_soak`).

## What could change the answer (future, not now)

- A **within-tick contrastive** target (rank co-occurring candidates against
  each other, not pooled win/loss) rather than a pooled P(win) regressor.
- **Clean per-trade cost labels** (`MB-20260629-ALLOC-COSTCAP`) so the target is
  true net-R, and a **correlation feature** (`MB-20260629-ALLOC-CORR`) so the
  selector sees correlated exposure — both are the substrate a *portfolio*
  (not per-candidate) selector would need.
- Re-run when the live real-money opportunity set is net-positive; a learned
  selector allocating a losing pie can only ever lose less.
