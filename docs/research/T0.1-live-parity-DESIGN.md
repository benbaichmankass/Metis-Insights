# M19 T0.1 → shadow — live-parity dependency design

> **Status:** DESIGN / decision memo (2026-07-01). T0.1 proved (offline,
> purged-CV, multi-symbol) that a frozen `chronos-bolt-tiny` embedding lifts the
> BTC/ETH-15m volatility-regime heads. This doc scopes the ONE gate standing
> between that offline result and a shadow-stage head that *observes live*: how
> the embedding gets computed at serve time without dropping a heavy
> `torch`+`chronos` dependency on the live money-box. **Proposes, decides
> nothing live** — promotion to shadow remains operator-gated. Evidence:
> [`T0.1-embedding-AB-evidence-2026-07-01.md`](T0.1-embedding-AB-evidence-2026-07-01.md)
> + [`T0.1-embedding-followup-evidence-2026-07-01.md`](T0.1-embedding-followup-evidence-2026-07-01.md).

## The question

A model at `candidate` is offline-only — its features are built by the trainer,
never live. Promoting it to `shadow` means it **predicts on every live bar**, so
its feature vector must be computed **live**, and — the load-bearing rule — the
live feature computation must match the offline one bar-for-bar
(`train == serve` parity, guaranteed in this codebase by *reusing the same pure
functions*, not re-deriving them; see `src/runtime/regime_shadow.py` +
`src/runtime/cross_asset_live.py`).

For every existing shadow head that parity is free: the features are pandas/NumPy
over OHLCV (`volatility_estimators`, `cross_asset_features`), and the live box
already has NumPy/pandas/LightGBM. The `tsfm_emb_*` block is different — computing
it live means running `chronos-bolt-tiny`, which needs **`torch` + `chronos-forecasting`**. Today those are **trainer-side only** (`requirements-backtest.txt`);
the live `requirements.txt` has neither. So the question is narrow and concrete:

> **Where does the Chronos forward-pass run at serve time, and how does its
> output reach the live regime head — without putting `torch` on the money-box?**

## What the recon established (the dependency boundary today)

- **Live `requirements.txt`: no `torch`, no `chronos`.** The live inference path
  is LightGBM-only — models ship as a **text booster string** (`lgb.Booster(model_str=…)`), zero heavy ML runtime. Confirmed 2026-07-01.
- **No ONNX anywhere in the repo.** No `torch.onnx`, `torch.export`, `skl2onnx`,
  no export workflow. An ONNX path would be greenfield.
- **The per-bar scorer is the natural integration point.**
  `src/runtime/regime_bar_scoring.py` already fetches candles once per
  `(symbol, timeframe)` group per tick, behind a **wall-clock fetch-gate**
  (fetch ~once per bar, not per tick) and a **per-tick wall-clock budget**
  (`REGIME_BAR_SCORING_BUDGET_S`, cold-start-safe). Any live embedding
  computation slots in exactly where `cross_asset_live.compute_live_cross_asset_row()` already merges an offline feature block into the row —
  same late-bound, fail-permissive, group-gated pattern.
- **`torch` is heavy enough it isn't kept resident even on the trainer.** On
  2026-07-01 no trainer venv had `torch` importable (the T0.1 install was
  ephemeral), and the trainer is a **1 OCPU / 6 GB** box. The live trader is
  **2 OCPU / 12 GB**, already running the pipeline + web-api + IB reach; the
  Ampere free-tier pool is full (no headroom to grow the VM).

## Latency is not the binding constraint

`chronos-bolt-tiny` is 9M params, CPU, and the embedding is needed **once per
closed bar** — for a 15m head that's once per 900 s. Even a pessimistic
100–300 ms CPU forward-pass is ~0.01–0.03 % of the bar interval, and the
existing fetch-gate already guarantees it runs ~once per bar, not per tick. The
per-tick budget defers any overrun to the next tick intact. So compute time does
**not** decide this. (A precise CPU figure was deferred — measuring it required
installing `torch` on the trainer just for a confirmatory number, and the
footprint evidence below is the actual decision driver. The number is worth
capturing before a final go, but it will not change the ranking.)

The real axes are: **(1) dependency footprint / memory on a full 2-core box,
(2) operational complexity + failure surface, (3) `train == serve` parity, (4)
build effort.**

## Options

### A — Install `torch` + `chronos` on the live VM, compute inline
Add the deps to live `requirements.txt`; call `chronos_embed_fn()` inside the
per-bar scorer.

- **+** Simplest wiring; reuses the exact offline `embed_fn` → parity is trivial.
- **−** Puts a **~1 GB+ torch install** + its import-time cost on the 2-core/12 GB
  money-box whose Ampere pool has no headroom. Import-time crash risk (the T0.1
  torch/Python-rc saga) now lands on the *live* box. Enlarges the live attack/upgrade surface for a shadow (observe-only) feature. Worst fit for the
  constraint the whole question is about.

### B — Sidecar embedder service (separate local process)
A tiny long-lived process on the live VM (or the gateway VM) holds
`torch`+`chronos`; the scorer calls it (`POST /embed {windows}`) and merges the
returned vector.

- **+** Isolates the torch footprint from the trader process; restart/version it
  independently; the trader stays LightGBM-only.
- **−** New always-on service to deploy, monitor, health-check, and resource-cap
  on an already-full box; a network hop + failure mode; parity depends on the
  sidecar running the same pinned `embed_fn`. Real ops weight for a shadow.

### C — ONNX export, ONNX-Runtime on the live VM
Export `chronos-bolt-tiny`'s encoder to ONNX once (trainer-side); the live box
runs `onnxruntime` (far lighter than torch) to embed.

- **+** Removes `torch`/`chronos` from live; `onnxruntime` is a smaller, stabler
  CPU dep. Compute stays inline (no sidecar).
- **−** **Greenfield** (no export precedent here); freezes a specific Chronos
  graph at export time (re-export on any model change); parity now hinges on the
  ONNX graph matching the torch `pipeline.embed()` mean-pool exactly — must be
  validated numerically. Most engineering, but the cleanest *inline* end-state if
  embeddings become load-bearing.

### D — Trainer-computed embeddings, shipped via the existing model mirror ★
Do **not** compute embeddings live at all. The trainer already runs
`chronos_embed_fn()` (that's how T0.1's side-streams are built) and already
**publishes artifacts to the live VM** through `runtime_logs/trainer_mirror/`
(the same channel the models themselves arrive on). A small scheduled trainer job
computes the latest `tsfm_emb_*` vector per live symbol at bar cadence and
publishes it; the live per-bar scorer reads the **most-recent published
embedding** as an as-of feature (exactly the offline as-of-carry semantics) and
merges it into the row — **zero torch on the money-box, no new live service, no
ONNX.**

- **+** Reuses infra that already exists end-to-end (trainer has torch; mirror
  ships data to live; the scorer already does as-of feature merges). No new live
  dependency, process, or export format. Lowest-risk way to get a shadow soak
  *started*.
- **−** The live embedding is as fresh as the mirror cadence (minutes), not
  tick-exact — a **staleness** the offline as-of-carry already tolerates and
  which is acceptable for a 15m head in **shadow** (observe-only). A subtle
  parity caveat: the trainer must embed from the **same candle view** the live
  head scores against (same source/timeframe), else the shipped vector drifts
  from what an inline compute would produce — bounded by using the same
  `connector_for_symbol` fetch the scorer uses. Not a fit if a future stage needs
  tick-exact embeddings (then C).

## Comparison

| Option | Live torch? | New live service | Parity risk | Build effort | Fit for shadow soak |
|---|---|---|---|---|---|
| A install | **yes (~1GB)** | no | trivial | low | poor (footprint) |
| B sidecar | isolated | **yes** | low | medium | medium |
| C ONNX | no (onnxruntime) | no | **must validate graph** | **high** | medium (best inline end-state) |
| **D mirror-publish ★** | **no** | no | low (staleness only) | **low** | **best** |

## Recommendation

**Start with D (mirror-publish) to run the shadow soak; hold C (ONNX) as the
graduation path if embeddings prove load-bearing and a future stage needs
tick-exact serve.** Rationale: the immediate goal is to get an
embedding-enhanced BTC/ETH-15m regime head *observing live* to confirm the
offline lift survives real-time — that is a **shadow** activity where minutes-stale
embeddings are fine, and D delivers it with **no new live dependency, service, or
export format**. Only if the soak confirms the edge AND a later stage (advisory,
or a faster-cadence head) needs tick-exact embeddings does the ONNX investment
(C) pay off. **A is not recommended** — it loads the money-box for an
observe-only feature. **B** is the fallback if D's staleness turns out to matter
before C is built.

## Proposed phased path (each step operator-gated, backtest-gated as noted)

1. **Rebuild the regime head at the production `vol_threshold`** (v003 used
   0.003 / 11.7 % volatile — the shipped head differs; the numbers only transfer
   after a matched rebuild). Offline; no live change.
2. **Stand up the mirror-publish job (D):** a trainer-side scheduled task that
   embeds the latest window per live symbol and publishes `tsfm_emb_*` to
   `trainer_mirror/`; add a `compute_live_embedding_row()` in the scorer that
   *reads* it (mirroring `cross_asset_live`). Wiring is Tier-1/observe-only.
3. **Promote the head `candidate → shadow`** (operator gate) and soak — compare
   the live shadow track record to the offline expectation; watch drift (KS/PSI).
4. **Only on a confirmed soak:** decide `shadow → advisory` (the live-influence
   gate) and, if tick-exact serve is then required, build the ONNX path (C).

## Open questions / risks

- **Parity validation for D:** confirm the trainer's published embedding for a
  bar equals what an inline compute would produce from the live candle view
  (same connector, same `context_len`/`stride`, same reduction seed). A
  one-shot offline-vs-published diff test gates step 2.
- **Mirror cadence vs bar cadence:** the publish job must run at least once per
  bar per symbol; size it against the trainer's 1-core budget (cheap — one
  9M-model forward-pass per symbol per bar).
- **Production `vol_threshold` rebuild (step 1)** is a prerequisite for *any* of
  A–D to carry the T0.1 numbers to the live head.
- **A precise CPU embed-latency number** is still worth capturing before a final
  advisory go (it bounds the inline options A/C), but does not change this
  recommendation.
