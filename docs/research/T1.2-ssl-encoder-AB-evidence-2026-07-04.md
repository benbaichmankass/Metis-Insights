# T1.2 — SSL corpus-encoder downstream A/B: evidence (2026-07-04)

**Verdict: CLEAN NEGATIVE, replicated across two corpus widths.** The
self-supervised "reads-everything" corpus encoder (masked-reconstruction MAE
over the wide `corpus_panel`) produces an embedding block (`corpus_emb_0..15`)
that, fed into the BTC-15m vol-regime boosting head, **beats neither the
no-embedding baseline nor the frozen-Chronos T0.1 embedding** on the two
decision metrics (`macro_f1`, `f1_volatile`). Widening the corpus 13→28 series
improved the *encoder's own* generalization but did **not** transfer downstream —
if anything it made the head's majority-class bias worse. This is the **third
independent negative** on price-representation for this head (after T0.1
frozen-embeddings-marginal and T1.1 deep-TCN-negative).

## The gate

Decision rule (from the T1.2 design): the `corpus_emb` block must lift ≥1
downstream head's purged-CV metric **beyond BOTH** the no-embedding baseline
**AND** the frozen-Chronos T0.1 embedding. Head under test:
`btc-regime-15m-lgbm-*-pcv` (3-class regime: quiet / range / volatile), the same
head the T0.1 and T0.4 A/Bs used. All three arms trained on the **identical**
`market_features` dataset (same rows, same purged walk-forward folds), so the
only difference is the feature block:

- **base** — 7 hand-engineered features
- **emb** — base + 32 `tsfm_emb_*` (frozen chronos-bolt-tiny, T0.1)
- **corpusemb** — base + 16 `corpus_emb_*` (this encoder, T1.2)

## Run 1 — thin corpus (13 FRED series, `market_features` v005)

Encoder: reconstruction_loss 0.036, **val_loss 2.006** (overfit — the thin panel
is too narrow for the MAE objective to generalize), ONNX parity 6.2e-06.

| arm | macro_f1 | f1_volatile | accuracy | f1_range | weighted_f1 |
|---|---|---|---|---|---|
| base | 0.556 | 0.388 | 0.624 | 0.725 | 0.680 |
| **emb (T0.1)** | **0.609** | **0.422** | 0.702 | 0.797 | 0.746 |
| corpusemb (T1.2) | 0.584 | 0.303 | 0.778 | 0.865 | 0.784 |

`corpus_emb` macro_f1 beats base (+0.028) but **loses to the T0.1 emb (−0.025)**;
`f1_volatile` is **worse than both** (base −0.085, emb −0.119). Higher
accuracy/weighted_f1/f1_range is a **majority-class artifact** — the embedding
pushes the head into the dominant "range" class and crushes the minority
"volatile" recall the vol-gate keys on.

## Run 2 — widened corpus (28 FRED series, `market_features` v006)

Corpus widened 13→29 series (PR #5539; `WILL5000INDFC` 404'd → 28 registered via
the per-series-resilient fetch): fuller Treasury curve (3mo/1y/5y/10y + 3m10y +
fed funds), inflation breakevens (5y/10y), VIX + Dow, IG credit OAS, broad-dollar
index + USDCAD/AUDUSD/USDCHF. Encoder retrained on the wider panel.

Encoder: reconstruction_loss 0.031, **val_loss 1.294** (down from 2.006 — breadth
**did** cut the encoder's overfit ~35%, exactly the diagnosed lever), n_series 28,
ONNX parity 1.7e-05.

| arm | macro_f1 | f1_volatile | accuracy | f1_range | weighted_f1 |
|---|---|---|---|---|---|
| base | 0.559 | 0.389 | 0.628 | 0.728 | 0.683 |
| **emb (T0.1)** | **0.611** | **0.423** | 0.705 | 0.799 | 0.748 |
| corpusemb (T1.2) | 0.584 | **0.269** | 0.825 | 0.898 | 0.812 |

`corpus_emb` macro_f1 essentially flat (0.584) — still below the T0.1 emb (0.611)
— and **`f1_volatile` got even *worse* (0.269 vs 0.303 thin)** while
accuracy/f1_range climbed further (0.825 / 0.898). Base and emb barely moved
between v005 and v006 (they don't read `corpus_emb`; the ~0.002 drift is fold
noise) — a clean sanity check that only the corpus arm changed.

## Why it fails (diagnosis)

1. **Majority-class bias, not signal.** The corpus embedding raises overall
   accuracy by leaning harder into the dominant "range" class, at the direct
   expense of the rare "volatile" class — the *opposite* of what a vol-regime
   gate needs. Widening the corpus amplified this (f1_volatile 0.303→0.269) rather
   than fixing it.
2. **Daily macro breadth ≠ intraday volatility signal.** The corpus is daily
   rates/FX/credit/equity context. That's real cross-asset structure (the encoder
   reconstructs it better with more series — val_loss fell), but it carries
   *risk-regime backdrop*, not the 15-minute volatility-burst signal the head is
   trying to classify. The frozen-Chronos T0.1 embedding, computed directly on the
   *bar window*, is a better vol-regime feature precisely because it sees the
   intraday series.
3. **Encoder generalization did not transfer.** A better-generalizing encoder
   (val_loss 2.0→1.3) produced a *worse* downstream head — the encoder learned to
   reconstruct the daily panel, but the reconstructed representation isn't
   aligned with the intraday-volatility target. This is the SSL transfer gap, and
   more corpus width doesn't close it here.

## Conclusion

Breadth was the hypothesized lever for price-representation (the M19 thesis:
"price-only representation is exhausted; the untested frontier is breadth"). We
built the SSL corpus encoder, widened the corpus, and A/B'd it under purged-CV
across two corpus widths. **It is a clean negative on the BTC-15m regime head** —
the "reads-everything" embedding beats neither the simpler frozen-TSFM embedding
nor the no-embedding baseline on the decision metrics, and corpus width does not
move it.

This does not condemn the corpus store or the SSL machinery — both are sound and
reusable (`corpus_panel`, `ssl_corpus_encoder`, the fail-closed ONNX parity gate,
the CPU-serve path). It condemns the *specific hypothesis* that a daily-panel SSL
embedding lifts the intraday vol-regime head. The encoder stays at **candidate**
(offline, refused by the live shadow factory); nothing here influences an order.

### What's left / where the value might still be

- **Not the regime head.** Two negatives (thin + widened) on BTC-15m regime is
  decisive for this head. Do not re-run it with more corpus width.
- **A task-matched target could still pay off** — the corpus embedding might help
  a head whose target is *itself* daily/cross-asset (a daily direction/risk head,
  or the M18 cross-sectional ranker over a multi-symbol daily universe), where the
  representation and the label live on the same clock. That is a *new* head, not a
  re-run, and is logged to the ml-review backlog rather than pursued here.
- **Light encoder regularization** (dropout / weight-decay / smaller embedding)
  is unlikely to change the downstream verdict — the failure is a
  representation-target *mismatch*, not encoder capacity — so it is not worth a
  burst. Logged, not pursued.

On the trainer VM (off-VM build host; `ICT_OFFVM_BUILD_HOST=1`):

```
# corpus already fetched into runtime_logs/trainer_mirror/corpus (28 series)

  --source fred_corpus_offvm --symbol-scope CORPUS --timeframe 1d \
  --version v002 --overwrite corpus_root=runtime_logs/trainer_mirror/corpus
python -m ml train ml/configs/corpus-ssl-encoder-mae-v1.yaml     # → val_loss 1.29
python -m scripts.ml.build_corpus_embeddings \
  --panel datasets-out/corpus_panel/CORPUS/1d/v002 \
  --model-state <newest corpus-ssl-encoder-mae-v1 model_state.json> \
  --out datasets-out/corpus_embeddings/CORPUS/1d/v002

  --version v006 --source datasets-out/market_raw/BTCUSDT/15m/v002 \
  --symbol-scope BTCUSDT --timeframe 15m --overwrite \
  market_raw_path=datasets-out/market_raw/BTCUSDT/15m/v002 \
  embedding_path=datasets-out/embeddings/BTCUSDT/15m/v001 \
  corpus_embedding_path=datasets-out/corpus_embeddings/CORPUS/1d/v002 \

python -m ml train ml/configs/btc-regime-15m-lgbm-emb-pcv-v1.yaml
python -m ml train ml/configs/btc-regime-15m-lgbm-corpusemb-pcv-v1.yaml
# read macro_f1 / f1_volatile from each run's metrics.json
```
