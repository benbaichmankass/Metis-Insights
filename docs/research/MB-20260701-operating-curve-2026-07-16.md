# MB-20260701-001 — BTC-15m vol-gate operating curve (gate 2, pinned) (2026-07-16)

## What this is

Gate 2 of the MB-20260701-001 Tier-3 gate chain: with the threshold→base-rate mapping
reconciled (gate 1, `MB-20260701-mapping-reconcile-2026-07-16.md`), **pin the operating
curve** — the live-faithful BTC-15m LightGBM regime head trained at each *genuine*
`vol_threshold` under one controlled protocol, so the operating point is measured, not
inferred. This replaces the mislabeled vt004-pcv probe (which trained on a 0.003 `v004` dir).

## Method

A **matched-sibling triple** — `market_features BTCUSDT/15m` version dirs **v513 / v514 / v515**
= genuine **0.003 / 0.004 / 0.005** labels (16.0% / 8.4% / 4.6% volatile), built together on ONE
common data window (identical `forward_vol`; differ only in the label threshold). Three live-faithful
mirrors of the shipped `btc-regime-15m-lgbm-v2` head (same 7 features, 60d recency half-life),
each with `class_weight.volatile` matched to its **true** base rate (inverse-base-rate held constant:
6.2 / 11.9 / 21.7), **purged 5-fold walk-forward CV**, candidate/offline. Manifests
`btc-regime-15m-lgbm-vt00{3,4,5}-pin-v1`; trained on the trainer VM 2026-07-16.

## Result — the pinned operating curve

| vol_threshold | volatile base rate | f1_volatile | macro_f1 | precision_v | recall_v | accuracy |
|---|---|---|---|---|---|---|
| **0.005 (SHIPPED / live gate)** | 4.6% | **0.249** | 0.598 | 0.173 | 0.467 | 0.902 |
| **0.004** | 8.4% | **0.341** | 0.629 | 0.249 | 0.563 | 0.853 |
| **0.003** | 16.0% | **0.468** | 0.669 | 0.367 | 0.661 | 0.793 |

- **Monotonic** in both `f1_volatile` (0.249 → 0.341 → 0.468) and `macro_f1` (0.598 → 0.629 → 0.669)
  as the volatile label densifies.
- The **shipped 0.005 operating point reads f1_volatile 0.249** — confirming, on a clean matched-weight
  purged-CV measurement, that the live gate's head is **data-starved** (the volatile class is too rare at
  4.6% for the head to learn it; recall 0.47, precision 0.17).
- **The genuine 0.004 point is now measured for the first time under purged CV: f1_volatile 0.341 /
  macro_f1 0.629** (the earlier vt004-pcv "0.004" was a mislabeled 0.003 run, `f1_volatile` 0.438 which
  matches this curve's 0.003 arm at 0.468 within the class-weight difference — that arm used a mismatched
  weight; this curve's 0.003 arm with matched weight reads 0.468).

## Interpretation — what this does and does NOT establish (honest bounds)

**Establishes:** the shipped 0.005 vol-gate head is the *weakest* operating point of the three; a denser
volatile label yields a materially more discriminative head, monotonically. The MB-20260701-001 thesis
("0.005 is data-starved; ~0.004 is materially better") is **confirmed with genuine, matched-weight,
purged-CV measurements** — the 0.004 point (0.341) is a real ~37% relative lift in f1_volatile over 0.005
(0.249), and macro_f1 (less prevalence-sensitive) also rises.

**Does NOT establish** that 0.003 (or 0.004) is the *right* gate operating point. Two honest bounds:
1. **`f1_volatile` is mechanically easier at higher prevalence** — part of the monotone rise is the
   metric's prevalence sensitivity (the T1.1 label-sensitivity evidence flagged this). `macro_f1` rising
   (0.598→0.669) is the more meaningful signal, but it is still a *classifier* metric.
2. **The gate is a live money-routing switch, not a classifier.** `REGIME_ML_VERDICT_MODE=use` drives the
   BTC vol-gate off this head's `P(volatile)`. Which operating point the gate should use is decided by
   whether it **beats the shipped 0.005 gate on a walk-forward money A/B** (the same bar the 2026-06-27
   go-live cleared), not by classifier metrics alone. A denser label also changes *when* the gate fires
   (more bars labeled volatile → the OFF-cells gate more often), which has a direct money consequence that
   only the backtest measures.
