## What this run tested

The first A/B (BTC 15m, `time_aware_holdout`) showed a modest lift and closed
with three operator-picked follow-ups. This run executed all three:

1. **Confirm before believing** — re-run the A/B under **purged walk-forward
   CV** (not a single holdout) and on **more symbols/timeframes** (ETH 15m, BTC
   1h) to rule out leakage/overfit and check it generalizes.
2. **Richer reduction** — does **past-only PCA** (256→32) extract more than the
   v0 **seeded random projection**?
3. **Harder head** — do the same embeddings lift the **direction** head
   (up/down/flat), where the hand-engineered features carry less signal?

All heads stay at `candidate` (offline). Every A/B fixes the dataset (v003) and
varies only the 32 `tsfm_emb_*` columns — the clean-comparison discipline the
first run's confound taught (see the prior doc's methodology note).

## Lever 1 — Confirm: purged-CV + multi-symbol

**Method:** `split_purged_walk_forward` (n_folds=5, label_horizon=5, embargo)
replaces `time_aware_holdout`; per-fold metrics are pooled and the model refit
on full data. Base (7 hand-engineered features) vs base + 32 `tsfm_emb`, same
dataset per symbol, same hyperparameters. Δ = emb − base.

| Symbol / TF | Δ macro_f1 | Δ f1_volatile | Verdict |
|---|---|---|---|
| **BTC 15m** | **+0.052** | **+0.035** | LIFT — holds under purged-CV, *larger* than the holdout's +0.041 |
| **ETH 15m** | **+0.058** | **+0.016** | LIFT — generalizes to a second symbol |
| BTC 1h | +0.017 | −0.001 | Weak / inconclusive (data-poorer at 1h; f1_vol flat) |

**Verdict — CONFIRMED.** The lift is not a holdout artifact: under the stricter
purged/embargoed walk-forward it is *at least as large* on BTC 15m (+0.052) and
**repeats on ETH 15m** (+0.058) — a different symbol the embedder was never
tuned for. BTC 1h is weak (macro_f1 nudges +0.017, f1_volatile is flat), which
tracks with 1h having far fewer bars than 15m — a data-quantity effect, not
evidence against the signal. The frozen, generic Chronos embedding carries
**volatility-regime** structure that the hand-engineered vol features don't
already capture, and it survives leakage-safe validation on the two liquid 15m
series.

## Threshold sensitivity — a volatile-base-rate CLIFF, not a smooth decay (added 2026-07-01)

Lever 1 (and the first A/B) ran at `vol_threshold=0.003` (v003), an **11.7 %**
volatile base rate. The **shipped** BTC-15m head labels volatility at `0.005` (a
**4.6 %** base rate). A go/no-go for promoting the embedding head to shadow has
to hold at the *production* threshold — so we swept the threshold under the same
purged-CV A/B:

| BTC-15m regime head, purged-CV | volatile base rate | Δ macro_f1 | Δ f1_volatile | base f1_volatile |
|---|---|---|---|---|
| **0.003** (research) | 11.7 % | **+0.052** | +0.035 | 0.38 |
| **0.004** | 8.4 % | **+0.037** | +0.031 | 0.30 |
| **0.005** (shipped head) | 4.6 % | **+0.012** | ~0.000 (flat) | 0.24 |
| **0.006** | 2.6 % | **−0.004** | −0.021 | 0.19 |

**Verdict — a clean MONOTONIC decay with the volatile BASE RATE: helpful →
marginal → slightly HARMFUL as the class gets too rare to learn.** Wherever the
volatile class is adequately populated (≥ ~8 %), the embedding delivers a solid
**+0.037–0.052 macro_f1** lift. It fades to +0.012 at 0.005 (4.6 %), and turns
mildly **negative** (−0.004) at 0.006 (2.6 %), where the base model itself is
nearly unlearnable (base f1_volatile 0.19) and the extra 32 columns add noise, not
signal. The effect is governed by the **base rate**, not the threshold per se —
it's the classic "a model can only exploit a signal where there's enough minority
class to learn it." The shipped head's 0.005 sits right in the marginal zone.
(What still survives at 0.005: volatile *precision* 0.16 → 0.20, recall traded
down — a hard gate benefits marginally.)

**Consequence for the roadmap — the promotion question sharpens, it doesn't
close.** Two mutually-exclusive cases:

1. **If the head's threshold is revisitable** — 0.004 gives BOTH a materially
   better head (f1_volatile 0.24 → 0.30) AND a clean embedding lift (+0.037). A
   lower-threshold embedding head is a genuine promote candidate. (Whether the
   gate's semantics permit 0.004 is a separate strategy question — the 0.005 was
   chosen as the gate's "volatile" operating point.)
2. **If 0.005 is locked by the gate** — embeddings don't help there; pivot the
   embedding effort to T0.2 / T0.4.

Either way, **the shipped head being weak at its own 0.005 operating point
(f1_volatile 0.24) is itself worth flagging to the ML track** — independent of
embeddings. Sweep points 0.006/0.007 (confirming the collapse persists at even
rarer base rates) were run short after the full sweep was preempted; see the
follow-up sprint log for the final curve.

## Lever 2 — Reduction: PCA vs random projection

**Method:** identical BTC-15m v003 A/B, but the 256-dim pooled embedding is
reduced to 32 by **past-only PCA** (basis fit on the oldest `pca_fit_frac=0.5`
of embeddings, then applied forward — leakage-safe) instead of the seeded
Gaussian random projection. Two comparisons: PCA-emb vs base, and PCA-emb vs
random-emb.

| Comparison | Δ macro_f1 | Δ f1_volatile | Reading |
|---|---|---|---|
| PCA-emb − base | **+0.0397** | **+0.0287** | PCA embeddings lift over base — same order as random (+0.041 / +0.030) |
| PCA-emb − random-emb | **−0.0013** | **−0.0009** | Essentially **TIED** — PCA gives no measurable gain over random |

(PCA-emb absolute: macro_f1 0.6159, f1_volatile 0.4046.)

**Verdict — NO GAIN; keep random projection.** PCA lifts over base by about the
same margin random projection does (+0.040 vs +0.041 macro_f1), and head-to-head
PCA is within ±0.001 of random on both metrics — noise, not a real difference.
The Johnson–Lindenstrauss intuition holds here: a seeded random projection
preserves the embedding's discriminative structure as well as a fitted PCA
basis, for **less** machinery (no fit step, no fold-safety bookkeeping, no
fit-fraction hyperparameter, deterministic from a seed). **Recommendation:
standardize on the random projection** as the T0.1 reduction. The PCA path stays
in the code (`reduction="pca"`) as a tested option, but it is not worth its extra
complexity.

## Lever 3 — Harder head: direction (up/down/flat)

**Method:** same embedding block fed into the BTC-15m **direction** head
(`btc-direction-15m-lgbm-{base-v003, emb-v1}`) instead of the regime head, same
dataset/hyperparameters, Δ = emb − base.

| Head | Δ macro_f1 | Reading |
|---|---|---|
| BTC-15m direction | **+0.004** | Negligible — within run-to-run noise |

**Verdict — NO EDGE for direction.** The embeddings that clearly lift the
volatility-regime head do essentially nothing for the directional head (+0.004
macro_f1 — a rounding-error move next to the regime head's +0.05). This is an
*informative* negative: it localizes where the frozen representation's value
lives. A generic time-series foundation model, mean-pooled over a trailing close
window, encodes **how the series is behaving** (calm vs violent, trending vs
choppy) — a volatility/persistence signature — far more than **which way it goes
next**, which is close to a martingale on liquid crypto and is exactly the thing
hand-features + labels can't crack either. **Do not invest T0.1 embeddings in
the direction head.** If a learned representation is ever to help direction, it
would need task-specific training (an in-house contrastive/masked encoder with a
directional pretext — the T1.2 program), not a frozen generic embedder.

## Consolidated verdict

| Question | Answer |
|---|---|
| Does the lift survive purged-CV? | **Yes** — BTC 15m +0.052 macro_f1 (≥ the holdout number). |
| Does it generalize across symbols? | **Yes on 15m** — ETH 15m +0.058. **Weak at 1h** (data-poorer). |
| Is PCA worth it over random projection? | **No** — tied within ±0.001. Keep random. |
| Do embeddings help direction? | **No** — +0.004. They carry volatility signal, not directional. |

**Net:** T0.1's core thesis is confirmed at the cheapest tier — a frozen,
$0-CPU, generic pretrained time-series embedding adds *real, leakage-safe* edge
to the **volatility-regime** heads on the liquid 15m series, and the simple
random projection is the right reduction. The value is specific (volatility
regime, 15m), not universal (not direction, weak at 1h) — which is a useful map
of where to spend the next effort.

## What this does / does not unlock

- **Does:** make the **volatility-regime** family the concrete candidate for the
  T0.1→shadow promotion path — a BTC-15m (and ETH-15m) regime head with the
  embedding block is now backed by leakage-safe, multi-symbol evidence.
- **Does not:** promote anything. The remaining gate items are unchanged from the
  first doc and are the real cost/benefit call:
  - **Live-parity dependency.** Shadow stage means the embedding is computed
    *live* (for train==serve parity), which puts `torch`+`chronos` on (or beside)
    the live money-box. That is the material cost to weigh against a
    +0.05-macro_f1 regime head — quantify it before proposing promotion (a
    trainer-side ONNX export / a sidecar embedder are the lighter-weight options
    to evaluate).
  - **Production `vol_threshold`.** v003's `vol_threshold=0.003` (11.7% volatile
    base rate) differs from the shipped head's threshold; a promotion path must
    rebuild at the production threshold before these numbers transfer to the live
    head.
  - **BTC 1h + ETH deserve their own shadow heads** before their cells rely on
    the ML vol label (cf. the `MB-20260628-VOLGATE-GOLIVE` per-symbol advisory
    requirement).

## Recommendation (proposed — operator picks)

1. **Standardize on random projection** for the T0.1 embedding block (done in
   code as the default; PCA retained as a tested-but-unused option). No further
   PCA work.
2. **Focus embeddings on the volatility-regime family, not direction.** Drop the
   direction-head embedding line of inquiry.
3. **If pursuing promotion:** scope the live-parity dependency cost (ONNX
   export / sidecar) as its own spike *before* committing — that, not the
   offline lift, is the decision. Otherwise T0.1 stands as a proven offline
   result and the effort moves to the T1.2 in-house encoder (the path that could
   help direction) or the wide-corpus data workstream.
