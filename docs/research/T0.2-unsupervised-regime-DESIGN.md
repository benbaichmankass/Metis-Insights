## Thesis

Every regime signal we ship today is **hand-bucketed**: a strategy is `range` or
`volatile` by a fixed `vol_threshold` on a realized-vol estimator, and the
LightGBM regime heads are *supervised to reproduce that hand label*. T0.2 asks
the unsupervised question: **if we let the data discover its own regimes, do the
discovered states carry more forward-useful structure than the hand threshold?**

Two things make this timely, not academic:

- **We already have the machinery.** `ml/trainers/causal_hmm_regime.py` fits a
  diagonal-Gaussian **HMM via GMM-EM** — it *discovers* K hidden states from the
  observation vector, then turns each state into a class posterior via a
  responsibility-weighted label map. The state discovery is unsupervised; only
  the final state→label projection uses the hand label. Its claimed edge over
  the tree heads is **temporal persistence** (the transition matrix smooths
  regime flicker) + **interpretability** (each state is a Gaussian vol regime).
  A 1h manifest (`btc-regime-1h-hmm-v1`) was authored (S-MLOPT-S14) but **never
  run to a purged-CV conclusion** — T0.2 finishes that AND extends it.
- **T0.4 gives the HMM a better observation.** The current HMM observes only
  *backward* vol (`yang_zhang_vol` + `rolling_log_return_vol`). T0.4 just proved
  the frozen-TSFM **forward** vol estimate (`fc_range_rel`) is the strongest
  offline vol-regime feature we have, base-rate-robust. Feeding `fc_range_rel`
  into the HMM's observation vector is the natural T0.2×T0.4 cross — a
  *forward-aware* state-discovery model.

## The evaluation — what "a discovered regime is useful" means

Discovery is cheap; **usefulness** is the bar. Two complementary reads, both
leakage-safe (filtered/forward-only posterior, purged+embargoed WF-CV):

1. **Supervised proxy (comparable to the whole M19 line):** map discovered
   states → `regime_label` and score macro_f1 / f1_volatile vs the LightGBM
   regime champion (`btc-regime-15m-lgbm-v2` / `-base`) on the SAME dataset +
   split. Does the HMM's persistence beat the tree head at reproducing the
   regime, and does adding `fc_range_rel` to the observation help?
2. **Unsupervised usefulness (the real T0.2 question):** do the discovered
   states separate a *held-out forward outcome* the hand label doesn't get to
   see — forward realized-vol dispersion, forward |return| — better than the
   hand `vol_threshold` bucket? (Silhouette / forward-outcome ANOVA on OOS
   folds.) A discovered regime that predicts forward dispersion better than the
   threshold is the thing worth promoting later.

Per the standing **"illusion of regimes" dissent**: a discovered regime earns
nothing until it adds OOS edge under purged-CV. A negative is an acceptable,
documented outcome — exactly as the direction-head test was (fc/emb both flat on
direction; the value is vol-regime-specific).

## Gates / discipline

- **Offline/`candidate` only.** No shadow promotion (Tier-3, operator-gated); no
  paid compute. The HMM predictor is stdlib-light (filtered forward recursion),
  but live per-bar scoring additionally depends on feature-parity for its
  observation columns (`yang_zhang_vol` live; and `fc_range_rel` would need the
  T0.4 live-parity/mirror-publish path) — a *later*, separately-gated step.
- **Leakage-safe by construction:** filtered (forward-only) posterior, never
  Viterbi/smoothing; purged+embargoed WF-CV; the leakage gate forbids
  `forward_log_return` / `forward_log_return_vol` / `regime_label` as features.
- Every A/B fixes the dataset and varies only the model / observation vector.

## Results (ran 2026-07-01, trainer VM) — DOCUMENTED NEGATIVE

Ran the 15m A/B at BOTH the shipped 0.005 (v520, 4.6% volatile) and the populated
0.003 (v513, 16.1% volatile) thresholds — LightGBM base head (control) vs the
Gaussian-HMM vs the forward-aware HMM, purged walk-forward CV on the fixed
dataset. Δ = arm − base.

| threshold | LGBM base (control) | HMM v1 | HMM fc-v1 |
|---|---|---|---|
| **0.005** (4.6% vol) — macro_f1 / f1_vol | 0.5908 / 0.2417 | 0.4908 / **0.000** | 0.4908 / **0.000** (≡ HMM) |
| **0.003** (16.1% vol) — macro_f1 / f1_vol | 0.5600 / 0.3905 | 0.4975 / 0.071 (recall 5%) | 0.4621 / **0.000** |

**Verdict — the HMM LOSES to the class-weighted LightGBM head at both thresholds.**
Δmacro_f1 is **−0.10 @ 0.005** and **−0.06 (HMM) / −0.10 (HMM-fc) @ 0.003**. Three
findings:

1. **Rare-class washout.** At 0.005 the HMM fully **collapses to an all-`range`
   classifier** (f1_volatile 0.0, recall 0.0; the 0.964 "accuracy" is just the
   95.4% range base rate). Its state→label projection is an **unweighted argmax**,
   so when volatile is 4.6% *every* discovered state's majority label is `range`.
   The LightGBM control avoids exactly this via `class_weight: {volatile: 28}`.
2. **Even at a populated base rate it underperforms.** At 0.003 (16% volatile) the
   plain HMM *barely* surfaces the class (f1_vol 0.071, recall 5%) — the argmax
   finally flips a state or two — but it's still far below the tree head
   (0.4975 vs 0.5600 macro_f1). A 3-state Gaussian HMM over 2–3 vol features is
   simply a weaker model than a class-weighted boosted tree on the same features.
3. **The forward forecast HURTS the HMM (the T0.2×T0.4 cross fails).** Adding
   `fc_range_rel` to the Gaussian observation vector made it *worse*, not better —
   at 0.005 it's byte-identical to the plain HMM (both collapsed), and at 0.003 it
   **fully collapses** (f1_vol 0.0, macro_f1 0.462 < plain HMM 0.498). The extra
   observation dimension (different scale/support) destabilizes the diagonal-
   Gaussian EM into a worse local optimum. So the forecast that lifts a *tree*
   head does not transfer to the HMM's generative emissions.

**Net:** T0.2 as a straight swap is a **documented negative** — unsupervised
Gaussian-HMM regime discovery does not beat the hand `vol_threshold` + a
class-weighted supervised head here. This is exactly what the "illusion of
regimes" dissent anticipated (validate, don't assume). The **honest caveat** is
that the HMM's label projection has *no class weighting*, which is the one
mechanism that could give it a fair shot at the rare class — see the salvage
experiment below.

### Salvage result — class-weighted projection (ran 2026-07-01) — RESCUED to near-parity

Implemented the salvage (`ml/trainers/causal_hmm_regime.py`
`label_projection_class_weight` knob, default off → byte-identical; `balanced` =
inverse-frequency — merged PR #5371) and re-ran `btc-regime-15m-hmm-cw-v1` on the
same v520 (0.005) + v513 (0.003) datasets, purged-CV.

| threshold | LGBM base | plain HMM | **HMM-cw (balanced)** | HMM-cw Δ vs base |
|---|---|---|---|---|
| **0.005** — macro_f1 / f1_vol | 0.5908 / 0.2417 | 0.4908 / **0.000** | **0.5841 / 0.2282** | **−0.007 / −0.014** |
| **0.003** — macro_f1 / f1_vol | 0.5600 / 0.3905 | 0.4975 / 0.071 | **0.5573 / 0.3727** | **−0.003 / −0.018** |

**Verdict — the one-line fix RESCUES the HMM to near-parity with the tree head.**
Class-weighting the projection turns the collapsed all-`range` model (f1_vol
**0.0**) into one that **surfaces volatile properly** (recall 0.46 @ 0.005, 0.77 @
0.003) and **essentially matches** the LightGBM head: macro_f1 within **−0.007 @
0.005** and **−0.003 @ 0.003** — inside run-to-run noise. It does **not beat** the
tree, so it is not an automatic promotion candidate, but it is now a **legitimate
alternative regime model** with two properties the tree lacks:

- **Temporal persistence** — the transition matrix smooths regime flicker (the
  tree scores each bar independently), which could matter more on the *stability*
  of the live gate than the point macro_f1 suggests.
- **Interpretability + orthogonality** — each state is a Gaussian vol regime, and
  it's a genuinely different model class → a candidate for an **ensemble** with the
  tree head, not just a replacement.

So T0.2 lands as a **qualified positive**: unsupervised HMM regime discovery, once
class-weighted, is *competitive* (not superior) with the hand-`vol_threshold` +
class-weighted tree head. The salvage was the difference between "broken" and
"viable." The forward-aware `fc`-in-observation line stays **dropped** (a confirmed
negative — destabilizes EM).

   proved it: as a *feature* to the tree head, not an HMM emission.
4. **Then** move the effort to the wide multi-asset data corpus workstream (the
   long pole for the T1.2 in-house encoder). The map so far: the tree head over
   hand-engineered + forecast features is the thing to beat, and a generic
   unsupervised state model *matches but doesn't beat* it — so the next real lever
   is better *data* (the corpus) and a *task-specific* encoder (T1.2), not another
   off-the-shelf model.
