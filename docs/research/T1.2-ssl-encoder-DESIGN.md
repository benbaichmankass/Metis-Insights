## 1. The question this answers

> **Does an in-house self-supervised encoder over a WIDE multi-asset panel produce
> a market-state embedding that lifts a downstream boosting head beyond both the
> no-embedding baseline AND the frozen Chronos embedding (T0.1)?**

This is the frontier the rest of M19 pointed at. Two independent negatives —
**T0.1** (frozen Chronos embeddings: marginal, base-rate cliff) and **T1.1** (deep
TCN end-to-end over raw bars: clean negative) — establish that *price-only*
representation over a single instrument's own bars is exhausted. The untested
lever is **breadth**: a representation learned over the *cross-asset context* (the
`corpus_panel`: rates curve, VIX term structure, equity indices, FX, commodities,
credit) that no single-instrument model can see. Self-supervision means it needs
**no labels**, sidestepping the binding constraint of the whole program (~350 real
trades). This is the "reads-everything" model the operator envisioned.

## 2. The thesis (why this shape, not another)

- **Deep enters as an ENCODER feeding the boosting heads, not an end-to-end
  replacement.** T1.1 proved end-to-end deep loses to LightGBM on our label/data
  regime. The value of depth, if any, is a learned *feature* the tree consumes —
  exactly the T0.1 embedding pattern, but with a *task-tailored, breadth-aware*
  representation instead of a frozen generic one.
- **Self-supervised, not supervised.** No trade labels are used to train the
  encoder — it learns from the abundant *unlabeled* daily corpus panel. This is
  the only lever in M19 that is not label-bound.
- **GPU-train, CPU-serve** — identical contract to T1.1: train the encoder in a
  spot-GPU burst, export the encoder to ONNX, parity-gate it, and serve the
  embedding on the CPU money-box via `onnxruntime` through the mirror-publish
  path (the fc-serve pattern from T0.4/PR-1a·1b). **No torch on the money box.**

## 3. What exists to build on (cite, don't re-derive)

- **The corpus + panel (the pretraining input).** The standing corpus store
  (`ml/datasets/corpus_store.py`, `runtime_logs/trainer_mirror/corpus/`) holds
  ~18 keyless-FRED daily series across rates / VIX term structure / equity /
  commodity / credit / FX. **`ml/datasets/families/corpus_panel.py` (C3)** already
  assembles them into the aligned, **leakage-safe** (strictly-prior as-of,
  one-day-lag, forward-fill, no-backfill) **date × series matrix** — the encoder's
  direct input.
- **The GPU-burst tier (proven).** `scripts/ml/gpu_burst/` — ledger-gated,
  teardown-guaranteed, public-data-only pod builds; the T1.1 run trained + ONNX-
  exported + parity-gated + ingested a torch model end-to-end for ~$0.03.
- **The ONNX-export + CUDA/CPU parity gate** (`ml/trainers/onnx_export.py`) —
  fail-closed; reused verbatim.
- **The embedding-feature-block pattern** (`ml/datasets/embedding_features.py`,
  T0.1) — a per-bar derived vector, written to a side-stream and as-of joined into
  `market_features` via a `*_path` kwarg, one-day-lagged, `0.0` when absent. The
  T1.2 embedding is its sibling.
- **The CPU mirror-publish serve path** (T0.4 / PR-1a·1b: `publish_live_forecasts.py`
  + `forecast_live.py` + the `regime_bar_scoring`/`regime_shadow` merge) — the
  exact template for serving a per-day encoder embedding live with parity, zero
  torch on the money box.

**T1.2 extends this scaffold with:** one new SSL trainer/encoder, one embedding
producer (offline + live), and one new manifest — no rebuild.

## 4. Architecture — masked-reconstruction encoder first (recommendation)

Two candidate self-supervised objectives over the `corpus_panel` window:

| | **Masked reconstruction (MAE-style)** ✅ recommended first | Contrastive (SimCLR/TS2Vec-style) |
|---|---|---|
| Objective | mask a fraction of the (day × series) cells, reconstruct them from the rest | pull augmented views of the same window together, push others apart |
| Small-data behaviour | robust; a direct, low-variance reconstruction loss | needs careful augmentations + large negative sets; higher variance on a thin panel |
| Missing-cell handling | **native** — the panel already has `None` cells (a series before its start); masking is the same mechanism | awkward — augmentations over a sparse panel are ill-defined |
| Interpretability | the reconstruction error is a legible health signal | the embedding geometry is opaque |
| Serve | encoder trunk → ONNX, same as T1.1 | same |

**Recommendation: a small masked-reconstruction encoder first.** On a ~18–40-series
daily panel with genuine missingness, MAE is the lower-variance, missingness-native
choice; contrastive is a T1.2b follow-up only if MAE under-delivers.

**Shape (v0):** input = a window of the last `L` days of the aligned panel
(`L × F`, start `L = 64`, `F` = the selected series). Per-cell standardization with
**train-fold-frozen** stats (stored in `model_state`). A missingness mask channel
marks `None` cells (never a fabricated zero fed as signal). Encoder = a small MLP
or 1-D conv trunk (parameter-light; the panel is daily, so sequences are short) →
a bottleneck **embedding** of width `d` (start `d = 16`). Decoder reconstructs the
masked cells; loss = masked-cell MSE. At serve time only the **encoder trunk** is
exported; the decoder is discarded. The embedding for day `D` is the trunk's output
on the window ending at `D` — a market-state vector `emb_0..emb_{d-1}`.

## 5. The embedding feature block (offline + live)

Exactly the T0.1 embedding pattern, sourced from the SSL encoder instead of frozen
Chronos:

- **Offline** (`scripts/ml/build_corpus_embeddings.py`, new) — runs the exported
  encoder over the historical `corpus_panel`, writes a per-day `emb_*` side-stream;
  `market_features` as-of joins it via a new `corpus_embedding_path` kwarg
  (one-day-lagged like the panel, `0.0` when absent), feeding the LightGBM heads.
- **Live** (`corpus_embedding_live.py`, new) — the CPU serve twin of `forecast_live.py`:
  a per-day producer computes the embedding on the trainer (onnxruntime, no torch),
  mirror-publishes `emb_*` per the fc-serve path, and the live scorer merges it
  into a shadow head's row — zero torch on the money box, train==live parity by the
  same structural contract (same encoder, same panel window).

Because the panel is **daily**, the embedding updates once/day and is joined as-of
to intraday bars — a documented resolution note (§8), not a blocker.

## 6. The gate (reuse T1.1's; two baselines to beat)

Held to the same `evaluate_gates` / `compute_oos_edge` under purged walk-forward.
The T1.2 embedding must **lift ≥1 downstream head's purged-CV metric** vs BOTH:

1. the **no-embedding baseline** (the plain LightGBM head), and
2. the **frozen-Chronos T0.1 embedding** (the incumbent representation).

If it beats neither beyond run-noise → the frozen encoder is retained and T1.2 is a
documented negative (cheap, decisive — like T1.1). Candidate heads to test the
lift on: the BTC-15m regime head (the M19 through-line) and the parked M18
cross-sectional ranker (whose tabular-only version hit OOS-AUC ≈ 0.51 — embeddings
are the untested lever there).

## 7. Phasing

- [x] **P0 — the panel** (`corpus_panel`, C3): the leakage-safe date×series matrix. **Landed.**
- [ ] **P1 — the SSL trainer** (`ml/trainers/ssl_corpus_encoder.py`, torch, GPU-burst;
      + a `corpus_embedding_features.py` block): masked-reconstruction encoder →
      ONNX-export + parity gate (reuse `onnx_export.py`) → `model_state` bundle.
      Its own PR + tests (windowing/mask leakage-safety, lazy-torch import,
      parity fail-closed), local-validated on a CPU-torch venv like T1.1.
- [ ] **P1b — one armed GPU burst** (ledger-gated, ≤ a few $) pretrains the encoder
      on the full corpus panel; ingest at `candidate`.
- [ ] **P2 — the A/B**: build the `emb_*` block, join into a head's `market_features`,
      run the gate vs the two baselines (§6). **Decision point.**
- [ ] **P3 — live serve + soak** (*only if P2 wins*): the CPU mirror-publish embedding
      producer + live reader (fc-serve twin) → operator-gated candidate→shadow.
- [ ] **P4 — advisory** (*only after a clean soak*): Tier-3 operator + walk-forward.

## 8. Honest gaps & risks

- **Most likely a negative, again.** Given T0.1 + T1.1, the base rate for "a new
  representation beats the boosted tree" is low. The bet is specifically that
  *breadth* (cross-asset context) is the one dimension not yet tried — but it may
  not help the narrow BTC-vol-regime target. The gate is designed to kill it cheaply
  if so. The value is decisive falsification for a few GPU-dollars.
- **Corpus is still thin** (~18 series). A masked-reconstruction encoder over 18
  daily series is small; the panel widens automatically as the yfinance/stooq
  breadth adapter (sector ETFs, crypto breadth) lands, so P1b can wait for a wider
  panel or run early and re-run.
- **Daily-vs-intraday resolution.** The embedding updates daily; intraday heads
  join it as-of (one-day-lagged). This is a *slow* context feature, appropriate for
  regime/direction conditioning, not a fast intraday signal — a scope note, and the
  reason to test it on the regime head first.
- **Missingness.** Early panel dates have `None` cells (a series before its start).
  The mask channel + masked loss handle this natively; the encoder must never be
  fed a fabricated zero as if it were signal (mirrors the fc/xa fail-permissive
  contract).
- **Parity is the real engineering risk** (as in T1.1): a CUDA-trained encoder must
  reproduce on CPU-ONNX within tolerance (the fail-closed gate) AND the live
  per-day embedding must match the offline one (the fc-serve parity-diff pattern).
- **Look-ahead in pretraining.** The encoder is trained on windows of the
  already-leakage-safe `corpus_panel`, but the standardizer must be fold-frozen and
  the A/B must run under purged walk-forward so the *downstream* evaluation is
  honest even though pretraining is label-free.

## 9. Out of scope for this design

- The contrastive variant (T1.2b — only if MAE under-delivers).
- Any promotion past `candidate`; any live-order-path change; any paid spend before
  an operator go on the P1b burst.
- The bigger multi-task foundation encoder + RL policy layer (M19 Tier-2).
