# M19 T1.1 — deep sequence regime head (GPU-burst, CPU-serve) — DESIGN

> **Status:** DESIGN (docs-only, Tier-1). The GPU-burst *execution path* is proven
> end-to-end (issue #5459: launch → build public dataset on the pod → `ml train` →
> stream the trained bundle back → teardown → ledger, ~$0.044); what it trained was
> the **incumbent LightGBM** head. This doc specs the *next* increment: the first
> **deep (torch) sequence model** trained in a burst and the **one new piece of
> machinery it needs — ONNX export + CPU/GPU numeric-parity gate**. It is a
> **proposal, no spend, no live-path change.** Everything graduates observe-only
> through the existing `candidate → shadow → advisory` ladder; the shadow→advisory
> promotion stays the operator-gated live switch.

## 1. The question this answers

> **Does a deep sequence model over raw BTC-15m bars beat the class-weighted
> LightGBM regime head on the same purged-CV gate?**

This is the cheapest, most falsifiable use of the GPU tier. The M19 Tier-0 sweep
established that *every frozen off-the-shelf model class* (T0.1 Chronos
embeddings, T0.2 HMM) **matches-or-loses** to the boosted tree over
hand-engineered + forecast (`fc_*`) features. The two remaining levers that could
actually beat it — a **learned deep sequence model** (this doc) and an **in-house
self-supervised encoder** (T1.2) — both need more than the free 1-OCPU trainer.

T1.1 is the falsification test: one ≤1-GPU-hr run answers "does a deep model over
raw bars beat the incumbent tree *at all*?" A **clean negative** is a valuable,
cheap result — it says the corpus + SSL (T1.2) is the only path and we learned it
for ~$0.30. A **positive** is the first model to beat the incumbent and justifies
the tier immediately. Either way the answer is decisive and within one burst's
budget.

## 2. What exists today (grounding — cite, don't re-derive)

- **Incumbent to beat:** `btc-regime-15m-lgbm-base-pcv-v003` / `btc-regime-15m-lgbm-v2`
  — a `LightGBMMulticlassTrainer` over 7 hand-engineered features
  (`vol_bucket`, `rolling_log_return_vol`, `log_return{,_lag_1,_lag_2}`,
  `hour_of_day`, `dayofweek`), class-weighted 28× on the ~3.6%-base-rate
  `volatile` class, target `regime_label`. Trained on the `market_features`
  dataset family, BTCUSDT @ 15m, `version: v002`.
- **Trainer/predictor contract:**
  - `ml/trainers/base.py::Trainer(ABC)` — a trainer implements
    `fit(rows, config) -> Mapping[str, Any]` (the `model_state`) and declares a
    `PREDICTOR_CLASS: ClassVar[type[Predictor] | None]`.
  - `ml/predictors/base.py::Predictor(ABC)` — a predictor implements
    `predict(row) -> float` (classifiers return a probability in `[0,1]`).
  - `model_state` is whatever `fit` returns; the runner `json.dumps`es it, so it
    must be JSON-serializable (LightGBM embeds the booster as
    `booster_str = Booster.model_to_string()`). **A torch model's weights are not
    JSON-native** — this is the crux the parity gate addresses (§ 5).
- **Dataset:** `ml/datasets/families/` build `market_features` from public candle
  data (Bybit klines → `market_raw` → `market_features`); **no `trade_journal.db`,
  no labels-from-trades** — so it is *exactly* the burst tier's data contract
  (public code + public market data only). Feature rows carry `ts`, `symbol`,
  `timeframe`, the OHLCV-derived columns, and the `regime_label` target.
- **Promotion gates:** `ml/promotion/gates.py::evaluate_gates(...)` +
  `ml/promotion/oos_edge.py::compute_oos_edge(...)` — the same `gate-check` the
  whole M19 line uses (purged walk-forward, embargo, non-degeneracy,
  beats-baseline, OOS-edge, sample-sufficiency, cross-run stability). A deep head
  is held to the **identical** gate; no bespoke rubric.
- **Live scoring machinery:** `src/runtime/regime_bar_scoring.py` scores each
  shadow-stage regime head on its own `(symbol, timeframe)` bar cadence under a
  **fetch-gate + wall-clock budget** (`REGIME_BAR_SCORING_BUDGET_S`). A deep head
  serves through this *unchanged* — it is just another predictor on the shadow
  side-channel (`ml/shadow/factory.py`, `src/runtime/shadow_adapter.py`).
- **GPU-burst tier (proven):** `scripts/ml/gpu_burst/` — `runpod_burst.py`
  (launch → SSH → build → train → stream bundle → teardown), `ingest_bundle.py`
  (namespaced-`-gpuburst`, forced-`candidate` registry ingest), the ledger-gated
  `gpu-burst-train.yml` workflow. The bundle is
  `{run_dir, model_state, metrics, manifest}`.

**This increment extends that scaffold with:** one new trainer class, one new
predictor class, one new ONNX-export+parity step, and one new manifest. No
rebuild.

## 3. Architecture choice — TCN first (recommendation)

Two candidate deep architectures over a raw bar window:

| | **Temporal Convolutional Net (TCN)** ✅ recommended first | Lightweight Transformer |
|---|---|---|
| Params (our sizes) | ~0.2–2M | ~1–5M |
| Inductive bias | **Causal dilated convolutions** — locality + multi-scale receptive field, matches bar-sequence structure | Global attention — flexible but data-hungry, weak prior |
| Small-data behaviour | Robust; fewer params, strong regularization from weight-sharing | Overfits readily below ~10⁴–10⁵ labeled windows |
| Causality guarantee | Structural (left-padding only; a bar can never see its future) | Needs an explicit causal attention mask — easy to get wrong = leakage |
| CPU-serve cost | Cheap; a few conv layers per bar | Attention is O(L²); heavier per bar |
| ONNX export | Clean (conv/relu/linear all first-class ONNX ops) | Clean but larger graph; attention-mask edge cases |

**Recommendation: a small causal TCN first.** On financial series with limited
labels, gradient boosting still wins and deep models earn their keep mainly via
representation learning + more data (the M19 thesis). Of the two deep options the
TCN is the lower-variance, causality-safe, cheaper-to-serve bet — the right thing
to falsify against the incumbent *first*. The Transformer is the fallback if the
TCN shows a near-miss that more capacity might close (a T1.1b follow-up), not the
opening move.

**Shape (v1):** input = the last `L` bars (start `L = 64`, ~16h of 15m bars) of a
small numeric feature vector per bar (log-returns + range/vol proxies + the
cyclical time encodings the tree already uses — *not* the pre-bucketed
`vol_bucket`, since the point is to let the model learn its own representation);
3–5 dilated causal conv blocks (dilations 1,2,4,8,16), channel width ~32–64,
global-average-pool over time → a 3-class head (`range`/`volatile`/…matching
`regime_label`'s classes). Class-imbalance handled the same way as the tree
(weighted loss ≈ inverse base rate). Target the **same** `regime_label`, so the
A/B is apples-to-apples.

## 4. The torch sequence Dataset + trainer

### 4.1 Windowing (leakage-safe)

A new pure helper builds fixed-length **causal** windows from the `market_features`
rows, sorted by `ts` within `(symbol, timeframe)`:

- Window `i` = features of bars `[i-L+1 … i]`; label = `regime_label` of bar `i`.
- **No forward columns** ever enter the window (the same `_REGIME_FORBIDDEN` set
  the tree enforces: `regime_label`, `forward_log_return`,
  `forward_log_return_vol`). The trainer re-checks this at `fit()` time — a
  forbidden feature raises `ValueError`, mirroring `LightGBMMulticlassTrainer`.
- Per-window feature standardization uses statistics **frozen from the training
  fold only** (mean/std stored in `model_state` for serve-time reuse) — never
  fold-global, to keep purged-CV honest.

### 4.2 `ml/trainers/torch_sequence.py`

A new `TorchSequenceTrainer(Trainer)` implementing the ABC:

```python
class TorchSequenceTrainer(Trainer):
    PREDICTOR_CLASS = TorchSequencePredictor  # ml/predictors/torch_sequence.py

    def fit(self, rows, config) -> Mapping[str, Any]:
        import torch  # lazy — the module imports without torch on the money-box
        # 1. build causal windows (§4.1), freeze feature schema + standardizer
        # 2. weighted cross-entropy (class_weight like the tree), Adam, early-stop
        # 3. train on CUDA if available else CPU (identical math; CUDA only faster)
        # 4. EXPORT the trained net to ONNX bytes; run the CPU/GPU parity gate (§5)
        # 5. return model_state = {
        #      "trainer": "...TorchSequenceTrainer",
        #      "onnx_b64": base64(onnx_bytes),        # the CPU-serve artifact
        #      "class_labels": [...], "feature_columns": [...],
        #      "seq_len": L, "standardizer": {"mean": [...], "std": [...]},
        #      "params": {...}, "n_train": N, "parity": {"max_abs_diff": ...},
        #    }
```

Key properties:

- **`torch` is a lazy import inside `fit`** — the module (and any code path on the
  live VM) imports without torch present, so **no torch/CUDA ever lands on the
  money-box**. Torch exists only on the rented GPU pod during the burst.
- **The `model_state` is JSON-serializable** because the exported artifact is
  ONNX **bytes base64-encoded**, not a live torch object. This is what lets the
  existing bundle/registry/mirror plumbing carry a deep model with zero changes —
  the ingest, the `experiments-runs/…/model_state.json`, the trainer-mirror
  publish all treat it as opaque JSON, exactly as they treat the booster string.
- Determinism: seed torch + cuDNN deterministic flags; record the seed in
  `params`. (Full bit-determinism across CUDA/CPU is not guaranteed — that's the
  whole reason for the parity *gate* rather than an equality *assertion*.)

### 4.3 `ml/predictors/torch_sequence.py`

`TorchSequencePredictor(Predictor)` reloads the ONNX graph via **`onnxruntime`
(CPU provider)** — *not* torch — builds the last-`L`-bar window from the live
feature row history the regime scorer already maintains, applies the frozen
standardizer, runs one `session.run(...)`, and returns the `volatile`-class
probability. This is the CPU-serve half; it never imports torch.

## 5. The one new piece of machinery — ONNX export + CPU/GPU parity gate

A LightGBM booster is already CPU-native, so "trained on the pod" and "served on
the CPU money-box" are byte-identical and no export/parity step exists in the
codebase today. A torch model breaks that: it trains on CUDA (or pod-CPU) and must
serve via `onnxruntime` on the live CPU box. Floating-point kernels differ across
CUDA ↔ CPU ↔ ONNX, so we must **prove the served predictions match the trained
model within tolerance** before the model is allowed to influence anything.

**`scripts/ml/gpu_burst/onnx_parity.py`** (new), run *on the pod inside `fit`'s
export step:

1. Export the trained `torch.nn.Module` to ONNX (`torch.onnx.export`, fixed
   opset, dynamic batch axis, `L`-length input).
2. Take a held-out reference batch of `K` windows. Compute:
   - `y_torch` = the torch model's output (the trained reference), and
   - `y_onnx` = `onnxruntime` CPU-provider output on the same batch.
3. **Parity gate:** `max_abs_diff(y_torch, y_onnx) <= PARITY_TOL` (start
   `1e-4` on the probability, tuned empirically) **and** argmax agreement = 100%.
   Record `max_abs_diff` + a small sample into `model_state["parity"]`.
4. **On failure the burst returns no model** (the artifact is refused, ledger
   still records the spend, the issue comment reports the parity miss). A model
   that can't be reproduced on the serve target must never reach `candidate`.

This mirrors the T0.4 live-parity contract (compute-on-trainer, serve-via-mirror,
zero torch on the money-box) but adds the **numeric** parity proof that a
CUDA-trained graph demands. It is the load-bearing new capability of this
increment; everything else reuses proven plumbing.

> **Note:** the CI workflow's `--export-onnx --parity-gate` flags referenced in
> early spec drafts were **fictional** (no such `ml train` flags exist). This doc
> makes them real *as a trainer-internal export step*, not as `ml` CLI flags — the
> burst calls the ordinary `ml train <manifest>`, and the ONNX export + parity gate
> live inside `TorchSequenceTrainer.fit`, so the existing burst driver runs it with
> no new CLI surface.

## 7. Live-serve contract (CPU, connection-free)

Identical spine to T0.4:

- **Where computed:** the ONNX graph rides the existing model-mirror channel
  (registry-store → `publish_trainer_mirror.sh` → live Models page); the live box
  loads it via `onnxruntime` only.
- **When scored:** `regime_bar_scoring.py`'s per-`(symbol,timeframe)` cadence,
  under the fetch-gate + `REGIME_BAR_SCORING_BUDGET_S` wall-clock budget — a deep
  head is just another shadow predictor. Bounded per-tick cost; a slow model is
  deferred, never stalls the loop.
- **Dependency footprint on the money-box:** `onnxruntime` (CPU wheel, ~few MB, no
  CUDA) — assessed as acceptable, same class as the Chronos serve decision. **No
  `torch`.** If even `onnxruntime` is deemed too heavy, the fallback is the same
  Option-D used for T0.1/T0.4: compute the head's output on the *trainer* and ship
  it as a mirrored feature — but for a true per-bar deep head, on-box
  `onnxruntime` is the cleaner path and is the recommendation.

## 8. Graduation gates (reuse, don't reinvent)

The deep head is held to the **same** ladder and the **same** `evaluate_gates`:

1. **P1 — offline bake-off (this increment, ~1 GPU-hr, `candidate`).** Train in a
   burst, ONNX-export + parity-gate, ingest as
   `btc-regime-15m-tcn-v1-gpuburst` @ `candidate`. Compute the OOS edge vs
   `btc-regime-15m-lgbm-base-pcv-v003` under purged walk-forward
   (`compute_oos_edge`, primary metric macro_f1, embargo). **Go/no-go: does it
   beat the incumbent beyond run-noise?**
2. **P2 — shadow soak (operator-gated `candidate → shadow`).** Only if P1 wins.
   Auto-wires onto the regime scorer's shadow side-channel, accrues a live
   prediction track record + drift stats, and — critically — a **live-parity diff
   test** (the mirrored ONNX output vs a trainer-side reference on the same bars)
   confirms serve==train in production, not just at build time.
3. **P3 — advisory (Tier-3 operator + walk-forward gate).** The money switch —
   `shadow → advisory` — identical to every other head. Only at `advisory` does it
   influence an order package (via the regime router). **This doc proposes nothing
   past `candidate`.**

## 9. Honest gaps & risks

- **Most likely outcome is a clean negative.** Given ~350 real trades and a
  ~3.6%-base-rate target, a deep model over raw bars probably *does not* beat the
  class-weighted tree + `fc_*` forecast features. That is the expected, cheap,
  decisive result — and it re-points the roadmap at T1.2 (SSL encoder on the wide
  corpus), which needs *no labels* and is the real frontier. We spend ~$0.30 to
  learn it rigorously instead of guessing.
- **Parity is the real engineering risk, not accuracy.** A CUDA-trained graph that
  doesn't reproduce on CPU-ONNX within tolerance is a silent-wrong-serve hazard —
  hence the build-time gate *and* the P2 live-parity diff. The gate must fail
  closed (no model out) on any parity miss.
- **Windowing leakage.** Fixed-length *causal* windows (bar `i` sees only
  `[i-L+1 … i]`, never a forward bar), a fold-frozen standardizer, and the
  forbidden-feature check are the guards. Note the leakage axis is the **forward
  label horizon**, not the backward lookback: a test sample's input window reaching
  back into training bars is legitimate (past data predicting the present), so the
  purge/embargo that matters is the one already applied to `regime_label`'s
  forward window (`label_horizon` + `embargo_fraction`) — it drops boundary train
  rows whose forward label overlaps the test period. The deep window adds no *new*
  embargo requirement beyond that, but the standardizer must stay fold-local so
  test-fold statistics never bleed into training.
- **Serve dependency creep.** `onnxruntime` on the money-box is new surface; keep
  it CPU-only, pinned, and behind the same import-lazy discipline so a missing
  wheel degrades the head to "unavailable" (em-dash), never crashes the tick.
- **Determinism.** Cross-device float nondeterminism means we gate on tolerance,
  not equality; re-runs may shift metrics within run-noise — the cross-run
  stability gate (`_gate_cross_run_stability`) already accounts for this.

## 10. Phase checklist

- [ ] **P1a** — `ml/trainers/torch_sequence.py` + `ml/predictors/torch_sequence.py`
      + `scripts/ml/gpu_burst/onnx_parity.py` + `btc-regime-15m-tcn-v1.yaml`, with

- [ ] **P3** — *only after a clean soak:* Tier-3 operator + walk-forward

## 11. Out of scope for this design

- The lightweight-Transformer variant (T1.1b — only if the TCN near-misses).
- The in-house SSL encoder (T1.2) and the wide multi-asset corpus that feeds it
  ([`T0-data-corpus-DESIGN.md`](T0-data-corpus-DESIGN.md)).
- Any promotion past `candidate`; any live-order-path change; any paid spend
  before an operator go on the P1b burst.
