## 2. Strategic Recommendation on VWAP

### What the evidence says

Two properly-powered adaptive backtests (#1536: +0.36 R, #1550: +0.13 R) across 24 × 14-day windows (≈ 11.5 months of coverage) landed near flat. Policy tuning moved the needle ±5 R/year at best — nowhere near enough to explain or fix the live 89%-losing-longs symptom.

The skip-list (`weak-up/low`, `sideways/low`) is robust at n=3 each and should stay. The single active override (`strong-up/low @ 2.0σ`, n=6, 5/6 positive) is real and should stay. Everything else in the policy table is noise or too small-n to trust.

### Recommendations

1. **Stop policy tuning.** The evidence bar is set and documented in `vwap_policy.py`. Re-opening the policy table before accumulating n≥3 in the remaining low-n regimes is noise amplification. The only justified addition is `strong-down/low @ 0.8σ` (n=2, +10.73 R in #1536) — evaluate in the next wide sweep but do not add it yet.

2. **Pivot to strategy-parameter sweep.** The next session's primary goal is:
   - Vary `ENTRY_STD_THRESHOLD`: {0.8, 1.0, 1.2, 1.5}σ
   - Vary `SL_STD_MULT_DEFAULT`: {0.3, 0.5, 0.7}σ
   - Add long-vs-short split to the per-window aggregate (the live asymmetry is 10.9% long-WR vs 40.9% short-WR — the backtest does not surface this yet)
   - Lock the policy table from #1537 throughout (don't vary policy and params simultaneously)

3. **Do not wire `vwap_policy` live until parameters are stable.** Wiring the policy gate while the threshold is wrong adds complexity without fixing the root cause. The correct order is: validate params → update params (Tier-3) → wire policy live (Tier-3).

4. **Test session-anchored VWAP** (UTC midnight reset vs current rolling 100-bar) as a second-order investigation after the parameter sweep. Anchor drift may be a contributing factor to the long/short asymmetry in trending markets.

5. **VWAP reframe:** VWAP is one structural variable in a mean-reversion system. The entry σ-threshold, SL multiplier, exit path, and anchor together define the R:R contract. Do not treat the policy gate as the primary lever when the core contract may be miscalibrated.

## 4. Decision-Tier Rules

### Tier 1 — Claude may proceed and self-merge
- Docs, sprint logs, sprint plans
- Tests and test infrastructure fixes
- Cleanup and refactors that don't change live strategy behavior
- Backtest harness additions: new CLI flags, output fields, aggregator fixes
- ML training configs, manifest changes (`.yaml.disabled` toggles)
- ML registry changes at `shadow` stage only
- CI/workflow changes that don't affect the live order path
- Comment drift fixes and code comment updates

### Tier 2 — Claude may implement, must ping Ben with a merge/hold summary
- Deployment script changes (`deploy/`, `scripts/ops/`)
- Operator-actions workflow changes (`.github/workflows/operator-actions.yml`)
- Shadow predictor wiring changes (`Coordinator._get_shadow_predictors`)
- Changes to the shadow → advisory promotion gate logic
- Any change that touches runtime plumbing and could affect production behavior if wrong, but doesn't directly change signal logic

### Tier 3 — Claude must not merge without Ben
- Any change to `vwap.py`, `turtle_soup.py`, or `ict_scalp.py` module constants or signal logic
- Entry / exit / SL / TP logic changes in any live strategy
- Risk-cap changes that alter strategy behavior (not just pure guardrails)
- Wiring `vwap_policy` or the regime classifier into `build_vwap_signal`
- Any ML model promotion: `shadow → advisory → limited_live → live_approved`
- Dry-run-to-live promotion for any account
- Parameter changes in `config/strategies.yaml` that change trade behavior (as opposed to adding informational fields)
- Any strategy enabled/disabled flip

**Note on tier re-classification:** If a change is clearly in a lower tier but the context makes it feel Tier-3 (e.g. a test fix that happens to be near live strategy code), keep it at the lower tier and document the rationale. Don't escalate unnecessarily. Don't under-escalate for speed.

## 5. Session Continuity Design

### Files that hold state

| File | Purpose |
|---|---|
| `docs/sprint-plans/ROADMAP-2026-05-19.md` | This file — master sprint sequence and statuses |
| `docs/sprint-plans/CURRENT-SPRINT.md` | Single-file handoff: active sprint ID, status, last action, next action |
| `docs/sprint-logs/S-<ID>.md` | Completed sprint log (written at sprint close) |
| `comms/follow_ups.json` | Open follow-up items and FU resolution tracking |

### How to mark a sprint complete

1. Write a sprint log to `docs/sprint-logs/S-<ID>.md` following `docs/SPRINT-LOG-TEMPLATE-CANONICAL.md`
2. Update the sprint's status in this roadmap to `✅ COMPLETE` with the merge date
3. Update `CURRENT-SPRINT.md` to point to the next sprint
4. Add any new FU items to `comms/follow_ups.json`

### How a new session identifies the next sprint

1. Read `docs/sprint-plans/CURRENT-SPRINT.md` — this is the single source of truth for active state
2. Read the referenced sprint's section in this roadmap
3. Read the most recent sprint log for context
4. Check `comms/follow_ups.json` for FU items that are blocking or due

### How to avoid getting stuck mid-sprint when a PM session is needed

Write the following to `CURRENT-SPRINT.md`:

```
STATUS: BLOCKED_PM
SPRINT: <sprint-id>
WAITING_FOR_BEN: <one sentence describing the decision needed>
LAST_COMPLETED: <what was done before the block>
READY_TO_CONTINUE: <exact next action once Ben unblocks>
```

Then stop. Do not implement the Tier-3 change while waiting. Leave the implementation branch open but un-merged.

### How to resume a blocked sprint

Read `CURRENT-SPRINT.md`. If STATUS is `BLOCKED_PM` and Ben has given approval (check PR comments, Telegram, or issue comments), update STATUS to `IN_PROGRESS`, implement the `READY_TO_CONTINUE` action, then complete the sprint normally.
