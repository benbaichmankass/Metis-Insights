### Risk guards

Three hard risk guards live in `safe_place_order()` (orders.py:85–115). All three raise `ValueError`, which aborts the order unconditionally:

| Guard | Env var(s) checked | orders.py lines |
|---|---|---|
| Position notional | `MAX_POSITION_USD`, order price | 85–93 |

Two additional safety gates operate as soft blocks: `DRY_RUN` (orders.py:128, defaults to `"true"`) and `ALLOW_LIVE_TRADING` (orders.py:135, defaults to `"false"`). Both default safe.

Kill-switch: `HALT_FLAG_PATH = "/tmp/trader_halt.flag"` (pipeline.py:9). Checked at pipeline.py:499 before `safe_place_order` is reached.

### Strategy multiplexing

Controlled entirely by the `STRATEGY` env var read at `run_pipeline()` entry (pipeline.py:461). Only **one** strategy builder executes per tick:

```python
# pipeline.py:409
STRATEGIES = ["breakout_confirmation", "vwap", "ict"]   # multiplexed priority order

# pipeline.py:461–474  (run_pipeline dispatch)
if   strategy_name == "multiplexed": builder = multiplexed_signal_builder
elif strategy_name == "vwap":        builder = vwap_signal_builder
elif strategy_name == "breakout":    builder = breakout_model_signal_builder
else:                                builder = killzone_signal_builder  # default
```

`multiplexed_signal_builder` (pipeline.py:419–447) iterates `STRATEGIES` in order; first `side ∈ {buy, sell}` with `qty > 0` wins. Strategies that raise are logged and skipped. If all return flat, returns `side=none`.

**Gap (open — Audit PR 5):** `killzone` is not included in `STRATEGIES` (pipeline.py:409), so `STRATEGY=multiplexed` never tries it. The default single-strategy path uses `killzone` as the `else` branch (pipeline.py:474), so it is exercised in non-multiplexed mode.

**Update vs original audit:** `STRATEGIES` now includes `"ict"` (added during M9 sprint, pipeline.py:409). The original audit showed only `["breakout_confirmation", "vwap"]`.

**Original finding (Audit PR 2):** `CURRENT_DAILY_LOSS_USD` and `CURRENT_OPEN_POSITIONS` were never injected before `safe_place_order`, so the daily-loss and open-position hard caps silently passed on every tick.

**Resolution:** PR #64 (`sha: see merge commit`) added `src/runtime/risk_counters.py` and calls `inject_runtime_counters(settings, exchange_client)` at `pipeline.py:503` before the news gate and before `safe_place_order`. Counters are now populated from the exchange API (positions) and the SQLite trade journal (daily PnL) on every actionable-signal tick.

**Verification at `875bfcc`:** `pipeline.py:503` — `settings = inject_runtime_counters(settings, exchange_client)` — confirmed present. `tests/test_runtime_risk_injection.py` — confirmed present and passing.

`pipeline.py:409` now reads `["breakout_confirmation", "vwap", "ict"]`. The `ict_signal_builder` is present and wired into `_STRATEGY_BUILDERS` (pipeline.py:412–415). No unit test covers the `ict` path in the multiplexed runner. This is a minor gap; the builder is tested individually but the multiplexed dispatch to it is not explicitly tested.

### F4 — `status=simulated` string in original audit was incorrect

The original audit stated `DRY_RUN=true → status=simulated`. The current code (orders.py:131) returns `status=dry_run`. This was likely wrong at the time of the original audit (or changed between `5574efc` and the current code). Updated in this refresh.
